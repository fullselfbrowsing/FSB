# 51-06 Plan: Sensitive Consumer/Social and Blocked-Policy Triage

## Goal

Make denied, sensitive, and degraded app states visible at descriptor and app level so blocked or policy-sensitive rows cannot be confused with T1-ready coverage.

## Tasks

1. Generate terminal-state rows for every descriptor.
   - Record `blocked`, `bridge-needed`, `uat-needed`, `degraded-discovery-pending`, `guarded-fail-closed`, or `t1-ready`.
   - Preserve denied origins as blocked-policy rows.
   - Do not change denylist behavior.

2. Generate app-level readiness rollups.
   - Roll descriptor states up to app status.
   - Surface blocked, bridge-needed, UAT-needed, degraded, guarded, and ready states.
   - Keep app status informational; do not alter routing or invocation.

3. Repair search readiness drift for newly proven handlers.
   - Add Phase 51 Retool and Asana handler slugs to the `search_capabilities` readiness override.
   - Add a regression test so all handler-backed T1 rows from the readiness report are covered by search readiness annotations.

## Verification

- `node tests/t1-terminal-states.test.js`
- `node tests/backing-status-annotation.test.js`
- `node scripts/verify-t1-readiness-gate.mjs`

## Exit Criteria

- Denied rows remain blocked-policy.
- App-level status distinguishes blocked, bridge-needed, UAT-needed, degraded, guarded, and ready states.
- Retool and Asana handler rows no longer appear discovery-pending in search readiness overrides.
