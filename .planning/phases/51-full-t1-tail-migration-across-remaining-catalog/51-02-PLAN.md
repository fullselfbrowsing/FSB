# 51-02 Plan: Existing-Head and Low-Risk Same-Origin Read Bulk Port

## Goal

Shrink the Phase 51 tail with only rows that can be proven as executable T1b through static, same-origin, no-parameter GET recipes. This is the first bulk pass; it must not promote Pattern-D, GAPI, sensitive/payment, write, destructive, parameterized, or token-header-dependent rows.

## Tasks

1. Add a conservative generated-recipe pass.
   - Read the canonical descriptor corpus.
   - Skip current bundled handlers and root recipes.
   - Admit only `backing: dom`, `sideEffectClass: read`, `GET`, no-param rows.
   - Require a vendored source file importing an explicitly allowlisted same-origin helper.
   - Require exactly one static literal API endpoint.
   - Exclude denied origins, Pattern-D/GAPI candidates, and sensitive/payment/token-style reads.

2. Emit generated bundled recipes.
   - Write generated recipes under `catalog/recipes/generated/`.
   - Refresh `extension/catalog/recipe-index.generated.js` with recursive recipe loading.
   - Keep descriptors unchanged so search semantics and provenance stay intact.

3. Route generated recipes through the existing resolver.
   - Preserve the existing search recipe lookup.
   - Add a fallback scan over the bundled recipe index so generated recipes resolve as T1b without per-slug registry edits.

4. Add focused regression coverage.
   - Assert the batch includes known safe low-risk reads.
   - Assert duplicate head rows, Pattern-D candidates, CSRF/bearer helpers, and payment-style reads remain excluded.
   - Assert every generated recipe validates as a no-param same-origin GET.
   - Wire the test into `npm run validate:extension`.

5. Regenerate readiness and tail reports.
   - Confirm the worklist shrinks by the number of generated recipes.
   - Keep all remaining non-ready rows in explicit workstreams.

- At least one low-risk same-origin generated recipe batch is active in the extension catalog.
- The readiness report increases executable T1-ready rows without changing public API, storage, MCP surface, or runtime security walls.
- The Phase 51 worklist shrinks and still records every remaining non-ready row.
