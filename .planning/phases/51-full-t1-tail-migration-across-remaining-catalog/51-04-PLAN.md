# 51-04 Plan: Asana Same-Origin Pattern-D Carveout

## Goal

Shrink the Pattern-D tail without relaxing the separate-origin wall by promoting Asana's read descriptors that are proven same-origin on `https://app.asana.com/api/1.0`.

## Tasks

1. Add an Asana read head.
   - Target `https://app.asana.com`.
   - Activate only the 15 read descriptors in the Asana Pattern-D workstream.
   - Build `GET` specs under `/api/1.0/...` and call `ctx.executeBoundSpec`.
   - Use `same-origin-cookie`; do not read cookies, page globals, tokens, localStorage, or script the browser.
   - Reject logged-out or unexpected Asana error envelopes with `RECIPE_DOM_FALLBACK_PENDING`.

2. Wire the handler into runtime and validation.
   - Add Asana to `HEAD_HANDLER_MODULES`.
   - Import `catalog/handlers/asana.js` in the extension service worker.
   - Add the handler to recipe-path, readiness, port-contract, and origin-classification registries.
   - Keep descriptor JSON, MCP schemas, public APIs, storage, and consent semantics unchanged.

3. Keep Pattern-D/GAPI boundaries intact.
   - Treat Asana as a same-origin carveout from the coarse Pattern-D app bucket.
   - Do not enable Linear, Datadog, Jira, cloud consoles, GAPI, or any generic cross-origin bridge.
   - Preserve separate-origin negative controls and the Pattern-D/GAPI hold for rows without same-origin proof.

4. Regenerate evidence.
   - Package the extension to copy the handler into `extension/catalog/handlers/`.
   - Regenerate Phase 44 readiness and Phase 51 tail worklist artifacts.
