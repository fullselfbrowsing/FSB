# 51-03 Plan: Retool CSRF Same-Origin Read Head

## Goal

Promote Retool's conservative no-parameter GET reads from DOM/discovery tail to executable T1a using the existing same-origin bound fetch primitive and Retool's cookie CSRF pattern.

## Tasks

1. Add a Retool head handler.
   - Target `https://retool.com`.
   - Activate only the 16 no-parameter Retool read descriptors already selected for this wave.
   - Build GET specs under `/api/...` and call `ctx.executeBoundSpec`.
   - Use `csrfSource` from the `xsrfToken` cookie to set `X-Xsrf-Token`.
   - Reject logged-out or unexpected Retool response envelopes with `RECIPE_DOM_FALLBACK_PENDING`.

2. Wire the handler into the runtime and guards.
   - Add Retool to `HEAD_HANDLER_MODULES`.
   - Import `catalog/handlers/retool.js` in the extension service worker.
   - Add the handler to the recipe-path allowlist and exact head-handler count tests.
   - Keep MCP schemas, public APIs, storage, consent semantics, and descriptor JSON unchanged.

3. Keep the origin gate narrow.
   - Accept Retool's relative `/api` runtime base only for `FsbHandlerRetool`.
   - Require vendored `retool-api.ts` evidence for relative endpoint construction and the `xsrfToken`/`X-Xsrf-Token` CSRF pattern.
   - Do not broaden the generic same-origin or same-registrable-domain gates.

4. Regenerate evidence.
   - Package the extension to copy the new handler into `extension/catalog/handlers/`.
   - Regenerate the Phase 44 readiness report and Phase 51 tail worklist.
