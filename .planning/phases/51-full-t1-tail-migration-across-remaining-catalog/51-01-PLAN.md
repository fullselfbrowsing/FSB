# 51-01 Plan: Full-Tail Worklist Generator, Acceptance Gate, and Batch Scheduler

## Goal

Create the executable control surface for Phase 51: a generated worklist over all non-ready catalog-tail rows, a validation test that keeps it tied to the readiness matrix, and a first-wave scheduling rule for the autonomous porting passes.

## Tasks

1. Add `scripts/report-t1-tail-worklist.mjs`.
   - Read from `scripts/report-t1-readiness.mjs`.
   - Exclude current `t1-ready` and `t1-guarded-fail-closed` rows.
   - Preserve slug, app, service, side-effect class, readiness, route feasibility, origin class, and next action.
   - Assign a workstream and required proof to every tail row.

2. Generate Phase 51 artifacts.
   - Write `51-T1-TAIL-WORKLIST.json` with every tail row.
   - Write `51-T1-TAIL-WORKLIST.md` with counts by workstream, app, route feasibility, and origin class.

3. Add a focused validation test.
   - Assert worklist tail count equals readiness learn/discovery/blocked totals.
   - Assert ready/guarded rows are not included.
   - Assert blocked rows remain `blocked-policy`.
   - Assert actionable rows have non-blocked terminal targets.

4. Use the worklist to schedule later plans.
   - First wave: existing heads and low-risk same-origin reads.
   - Second wave: Pattern-D and GAPI bridge families.
   - Third wave: write/destructive UAT activation.
   - Parallel policy track: sensitive and denied rows.

## Exit Criteria

- Phase 51 is visible in the active roadmap and requirements traceability.
- The generated worklist covers the full current tail without overclaiming execution readiness.
- The worklist can shrink naturally as future T1 ports land.
