## Color

The section uses ONLY existing semantic tokens. The control panel's dominant/secondary/accent split is already established; this section conforms to it. Accent (FSB orange `#ff6b35` via `--primary-color`/`--fsb-primary`) stays reserved for primary affordances and the active segment — it is NOT used to signify danger or "sensitive."

| Role | Token | Value (light) | Usage in this section |
|------|-------|---------------|-----------------------|
| Dominant (60%) | `--bg-secondary` / `--bg-primary` | `--fsb-surface-base` / `--fsb-surface-elevated` | Page + card surfaces (inherited) |
| Secondary (30%) | card surface + `--border-color` | `--fsb-border-subtle` | `.settings-card` chrome, list/table borders, sidebar nav |
| Accent (10%) | `--primary-color` | `#ff6b35` | Primary "Grant" / "Export" CTA emphasis, active segmented-control segment (`.detail-btn.active`), nav-active marker |
| Warning (sensitive friction) | `--warning-color` | `#f59e0b` | Sensitive-origin badge fill + flag icon (banking/email/gov heuristic, D-14) — same hue already used by `.feature-badge.beta` (`#d97706`/amber family) |
| Destructive | `--danger` (= `--error-color`/`--fsb-danger`) | red | "Clear audit log" / "Deny" destructive action via `.control-btn.danger` / `.overflow-item.danger` |
| Info (pending) | `--info-color` | `#0891b2` | Pending-Ask queue count notice accent (matches `.sync-status-pill[data-state="remote-active"]`) |

Accent reserved for: the primary CTA emphasis (Grant, Export), the ACTIVE segment of the Off/Ask/Auto control, and the nav-active state. Accent is NEVER used for the sensitive flag (that is warning/amber) and NEVER for destructive actions (that is danger/red). Do not introduce a new orange/red/amber hex — use the tokens above.

Sensitive vs denylisted color contract:
- **Sensitive (heuristic, D-14):** warning/amber badge (`--warning-color`), origin row remains interactive but the mode control is constrained to Off/Ask (Auto disabled with a tooltip explaining the friction).
- **Denylisted (`service-denylist.json`, D-15/D-16):** danger/muted treatment — row greyed (`opacity` + `--text-secondary`), badge red (`--danger`), ALL controls `disabled` (non-enableable), with the `deniedReason` shown as the hint.

## Wiring Contract (options.js — mirror the Phase-245 pattern EXACTLY)

The section follows the `fsbChangeReportsEnabled` lifecycle (`options.js` defaultSettings → cacheElements → setupEventListeners → loadSettings → saveSettings), adapted because consent/audit state lives in dedicated stores (`consent-policy-store.js`, `audit-log.js`), not in `defaultSettings`.

1. **cacheElements** — add `elements.consentOriginList`, `elements.pendingRequestList`, `elements.auditLogTable`, `elements.auditExportBtn`, `elements.auditClearBtn` (`document.getElementById(...)`, same idiom as `options.js:127-218`).
2. **setupEventListeners** — wire the section the same way the nav already wires (`switchSection('consent-audit')` flows for free via the existing `elements.navItems.forEach` at `options.js:222-224`). Add: segmented-control click handlers (write mode to `consent-policy-store`), mutating-toggle `change` handlers, Grant/Deny click handlers, `auditExportBtn`/`auditClearBtn` clicks (mirror `clearLogs`/`exportLogs` at `options.js:564-570`).
3. **loadSettings / render** — on section enter (and on `chrome.storage.onChanged`), read the consent envelope + audit ring and render rows (mirror the live-counter refresh idiom at `options.js:1026-1054`; debounce storage-driven re-render ~100ms as the cap counter does).
4. **persistence** — consent changes write straight to the consent store and reflect immediately; NO global Save bar entry (these are not in `defaultSettings`). Audit Clear writes to the audit store after confirmation.
5. **import ordering** — add `<script src="consent-audit-ui.js"></script>` (or inline into `options.js` per planner) AFTER `options.js` in `control_panel.html`, alongside the existing `install-identity-ui.js` tail (`control_panel.html:1613-1627`). Any new store module reached from the UI is added as a `<script src>` before it is used.

## Per-Element States

### 1. Per-Origin Consent list
- **default:** each seen origin renders a `.setting-item` row: origin (left), Off/Ask/Auto segmented control + "Allow mutating (write) calls" toggle (right). New/unseen origins default to **Off**, mutating **off**.
- **mutating-disabled:** when mode == Off, the mutating toggle is visually de-emphasized + `disabled` (write is meaningless with no consent). It becomes enabled only when mode is Ask or Auto.
- **sensitive (D-14):** amber `Sensitive` badge after the origin; the `Auto` segment is `disabled` (Off/Ask only); hint shows the sensitive copy. Selecting Ask still forces per-action confirmation downstream.
- **denylisted (D-15/D-16):** row greyed; red `Blocked` badge; segmented control + toggle all `disabled` and non-enableable; hint shows `deniedReason`.
- **empty:** `.session-empty-state` with the "No origins seen yet." copy.
- **loading:** brief skeleton/placeholder text `Loading consent settings…` until the store read resolves (best-effort, like the cap counter).
- **error:** the store-load error copy in a `.setting-hint` styled with `--danger`; existing policies untouched.

### 2. Pending Requests queue
- **default:** `.session-list` of rows; each shows `{origin} · {slug} · {method} · {sideEffectClass}` and `Grant` (`.control-btn`) + `Deny` (`.control-btn.danger`). Newest first.
- **count notice:** a `.sync-status-pill`-style `N pending` indicator in the card header (info hue) when count > 0; hidden at 0.
- **mutating pending:** rows whose `sideEffectClass` is mutating carry an inline amber tag `write` next to the method so the elevated nature is visible before Grant (D-04).
- **grant:** writes consent for that origin/scope, removes the row, surfaces a `showToast('Access granted', 'success')` (reuse existing toast).
- **deny:** removes the row without granting; `showToast('Request denied', 'warning')`.
- **empty:** `.session-empty-state` with the pending empty copy.

### 3. Audit Log viewer
- **default:** read-only table (header row + redacted entry rows) ordered **newest first**; columns Time / Origin / Capability / Method / Side Effect / Consent / Outcome. Failed-outcome rows use the `.log-entry.error` tint. NO column for, and NO rendering of, args/tokens/cookies/CSRF/body — the schema (D-10) excludes them and `redactForLog` (D-11) is applied before persistence; the UI only ever reads the already-redacted ring.
- **export:** `Export Log` downloads the redacted entries as JSON (mirror `exportLogs`/diagnostics export). Filename e.g. `fsb-audit-log-{timestamp}.json`.
- **clear (destructive):** `Clear Log` opens a confirm (window.confirm or existing modal idiom) with the destructive-confirmation copy; on confirm, clears the audit store and re-renders to the empty state; `showToast('Audit log cleared', 'warning')`.
- **empty:** `.session-empty-state` with the audit empty copy.
- **loading:** placeholder row `Loading audit log…`.

### 4. Legal & Service Policy card
- **default:** a `.settings-card` with the legal body copy and the three links (legal posture, blocked services, privacy). Static; no state machine. The "View blocked services" target enumerates the `deniedOrigins` from `service-denylist.json` (rendered inline list or linked).

## Accessibility

- **Nav:** the new `.nav-item[data-section="consent-audit"]` is keyboard-focusable and activates on Enter/Space exactly like existing nav items (no regression to the shared nav handler).
- **Segmented control (Off/Ask/Auto):** implement as a labeled group — wrap in a container with `role="radiogroup"` and `aria-label="Consent mode for {origin}"`; each `.detail-btn` carries `role="radio"` + `aria-checked` reflecting the active mode; arrow-key navigation between segments; disabled segments use `aria-disabled="true"`. (Augments the existing `.detail-btn` markup which today is plain buttons.)
- **Mutating toggle:** native `<input type="checkbox">` inside `.modern-toggle` (label association via wrapping `<label>`), identical to `fsbChangeReportsEnabled`; gets an `aria-label` when its visible label is insufficient; `disabled` state mirrors the Off-mode constraint.
- **Badges:** `Sensitive` / `Blocked` badges are not color-only — they carry visible text AND an icon (`fa-triangle-exclamation` for sensitive, `fa-ban` for blocked) so the state is conveyed without relying on hue.
- **Pending Grant/Deny:** real `<button>`s with discernible names ("Grant", "Deny"); destructive Deny additionally carries `aria-label="Deny request from {origin}"`.
- **Audit table:** use a real `<table>` with `<thead>`/`<th scope="col">` for the seven columns so screen readers announce headers (the `.log-entry` styling can be applied to `<tr>`/`<td>`); table has `aria-label="Audit log"`.
- **Clear Log confirmation:** focus moves into the confirm; Escape cancels; confirm is reachable by keyboard.
- **Focus visibility:** rely on existing focus-visible styling already applied to `.control-btn` / `.detail-btn` / inputs — do not suppress outlines.
