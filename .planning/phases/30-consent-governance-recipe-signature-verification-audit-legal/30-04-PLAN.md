<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user action (options page) → consent store | The UI is the ONLY surface that grants consent from user intent; a grant must write consent for that exact origin/scope, never a global enable. |
| audit ring → audit viewer | The viewer must render only the already-redacted ring; it must not have a column or template that could surface a secret. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-30-14 | Information disclosure | a secret column/field in the audit viewer | mitigate | The audit table renders ONLY the seven redacted columns; the UI test asserts string-absence of args/body/cookie/token/csrf/authorization/bearer (GOV-06 at the UI layer, complementing the Plan-02 store redaction). |
| T-30-15 | Elevation | enabling a denylisted or over-friction origin via the UI | mitigate | Denylisted origins render non-enableable (controls disabled); sensitive origins disable the Auto segment (D-14) using classify, the SAME source of truth the gate enforces; the GATE (Plan 02 step 4) is the authoritative sensitive+Auto downgrade — the UI cannot grant past the denylist or silently enable Auto on a sensitive origin because the gate re-checks classify on every invoke. |
| T-30-16 | Tampering | grant writing a wrong/broad scope | mitigate | Grant writes consent for the exact pending origin/scope only; consent lives in its own store, never the global Save-bar; the pending record carries no secrets (read-only origin/slug/method/sideEffectClass). |
| T-30-SC | Tampering | npm/pip/cargo installs | mitigate | Zero packages — vanilla HTML/JS clone of the existing control-panel components; no registry, no new dependency. |
</threat_model>
