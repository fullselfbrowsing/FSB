<interfaces>
```js
FsbDelegationFeed.render(container, snapshot, { hydrated: boolean })
FsbDelegationFeed.renderEntry(entry)
FsbDelegationFeed.renderSummary(summary)
```

The side panel consumes only canonical background snapshots/runtime results. It never reads provider output, storage, registry, or bridge directly and never sets held/running/stopped before the authoritative response. One `aria-live="polite"` region announces new live entries; hydrated entries are inserted with announcements suppressed. Offline/disconnected alerts use `role="alert"`; events use semantic `article`, summaries use `dl` and native `details`.

The accepted live message is exactly Plan 02's `{ type:'FSB_DELEGATION_UPDATED', snapshot:DelegationSnapshot, announceSequence:number|null }`. Each panel/requested conversation tracks its exact server delegation id, rejects updates for every other active id plus extra keys/invalid nested shapes, renders only its matching `snapshot.entries`, and suppresses delivery duplicates only when the exact `(snapshot.delegationId, announceSequence)` was already rendered. Feed rows read client/profile/model/session/allowed-tools only from `entry.init`, call id/name/tab/args/status/duration only from `entry.tool`, retry class/attempt/max/delay only from `entry.retry`, summary values only from `metrics`, and terminal classification only from the closed terminal code. It never parses `title`/`detail` as machine data, repairs, deduplicates, or reorders stored ledger rows.
</interfaces>
