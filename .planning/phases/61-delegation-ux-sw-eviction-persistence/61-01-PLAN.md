The new provider-selection and consent contracts are closed:

```js
// Existing options/storage shape remains unchanged. API kind may carry the
// canonical empty string (or a latent canonical agent choice); no write-back.
{ providerKind: 'api' | 'agent', agentProviderId: '' | 'claude-code' }

FsbDelegationPreflight.check({ providerKind, agentProviderId, modelProvider, bridgeState })
// -> { ok: true, kind: 'api', providerId: <existing-api-id>, agentProviderId: '' }
//  | { ok: true, kind: 'agent', providerId: 'claude-code', providerLabel }
//  | { ok: false, code: 'agent_offline'|'agent_unpaired'|'unsupported_provider', providerId, providerLabel }

FsbDelegationConsent.issueChallenge({ providerId, taskDigest, ttlMs })
FsbDelegationConsent.consumeChallenge({ challengeId, providerId, taskDigest })
FsbDelegationConsent.getTrusted(providerId)
FsbDelegationConsent.writeTrustFromChallenge({ challengeId, providerId, trusted: true })
FsbDelegationConsent.clearTrusted({ providerId }) // authority-reducing, exact canonical id only
```

Challenges live only in `chrome.storage.session`, carry a schema version, nonce, issue/expiry timestamps, exact provider id/task digest, and a one-use trust-write marker, and are atomically removed by the sole service-worker authority before start. `writeTrustFromChallenge` may enable only the exact provider's versioned `chrome.storage.local` trust bit after validating the still-live challenge; it marks that challenge's trust-write slot used but cannot consume/grant start. `clearTrusted` is a separate idempotent authority-reducing operation: it accepts only the exact canonical provider id, serializes the local write to false/removal, never starts a run, and needs no spawn challenge. Plan 06 exposes it only through the Providers restore-confirmation command.
