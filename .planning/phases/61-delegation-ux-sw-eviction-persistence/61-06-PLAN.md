```js

{ type: 'FSB_DELEGATION_TAKE_CONTROL', delegationId: string }
{ type: 'FSB_DELEGATION_RESUME', delegationId: string }
{ type: 'FSB_DELEGATION_STOP', delegationId: string }
{ type: 'FSB_DELEGATION_SNAPSHOT', delegationId: string | null }
```

The exact emitted update is Plan 02's closed `{ type:'FSB_DELEGATION_UPDATED', snapshot: DelegationSnapshot, announceSequence: number|null }`; no extra keys or provider-native event are allowed. Its entries carry only the exact typed `init`, `tool`, `retry`, and `metrics` structures, and terminal state carries only `DelegationTerminalCode`; background/controller code never encodes or recovers those values through `title`/`detail`. Preflight and consent use Plan 01's closed results. Trust-enable success is exactly `{ ok:true, providerId:'claude-code', trusted:true }`; clear success is exactly `{ ok:true, providerId:'claude-code', trusted:false }`; trust failure is `{ ok:false, code:'trust_challenge_invalid'|'trust_provider_changed'|'trust_storage_failed'|'unsupported_provider' }`. Start/hold/resume/stop success is `{ ok:true, snapshot:DelegationSnapshot }`; failure is `{ ok:false, code:'invalid_request'|'delegation_mismatch'|'invalid_state'|'preflight_failed'|'consent_required'|'consent_invalid'|'start_rejected'|'mapping_unavailable'|'hold_failed'|'hold_lease_failed'|'resume_failed'|'resume_ownership_lost'|'stop_failed'|'persistence_failed', snapshot:DelegationSnapshot }`. Exact-key validation rejects every extra field.

The service worker creates exactly one controller per classloader, but that controller owns independent records keyed by every active server delegation id. Start ordering is config -> pure preflight -> authoritative trust reload -> challenge consume (or background-mint/consume for an already trusted provider) -> `delegate.start` acceptance/server id -> controller/session ledger/chat/UI mutation. `FSB_DELEGATION_SET_TRUST` is accepted only while its unconsumed challenge is live, only when a fresh authoritative config read still selects the same canonical agent provider, and it calls `writeTrustFromChallenge`; `trusted` on START or any caller consent boolean is forbidden. `FSB_DELEGATION_CLEAR_TRUST` accepts only the canonical provider id, calls the authority-reducing `clearTrusted`, is idempotent, requires no challenge, and cannot start or mutate a run. Hold ordering is controller active-tab query -> complete registry ownership snapshot -> supervisor hold confirmation -> registry seals the complete mapped set -> persisted held entry. Resume ordering is validate complete lease/tab identities -> restore complete ownership -> supervisor resume; any failure retains or re-seals if safe then cancels. Stop ordering is cancel -> tree settled -> exact mapping release -> terminal entry/count -> heartbeat release. Every command, observer event, watchdog, and result is routed by exact server id and cannot settle another active record.
