# Phase 34 — Research / Implementation Map

## Mechanism
CDP `DOM.setFileInputFiles({nodeId, files:[absPath]})` is the only API that populates `<input type=file>` from a disk path (page JS is forbidden). Reuse the existing `chrome.debugger` attach/`sendCommand`/detach seam (`background.js` `executeCDPToolDirect` ~:14512, `handleCDPMouseClick` ~:14112).

## Critical architecture finding: two front doors, route as `background`
The MCP and autopilot front doors do NOT share a single CDP chokepoint. Make `upload_file` a `_route: 'background'` tool (like `execute_js`), so both converge on background-side handlers and share one CDP helper:
- **MCP**: `manual.ts` auto-registers every non-read-only `TOOL_REGISTRY` tool → wire `mcp:execute-action` → `mcp-bridge-client.js` `_handleExecuteBackground` (:869) → `dispatchMcpToolRoute` (needs a `MCP_PHASE199_TOOL_ROUTES` entry in `mcp-tool-dispatcher.js`). Tab ownership (`TAB_NOT_OWNED`) is enforced automatically by `resolveAgentTabOrError` + `checkOwnershipGate` before the handler.
- **Autopilot**: `tool-executor.js` `executeTool` → `executeBackgroundTool` switch (~:224). LANDMINE: `handleDataTool` is `null`, so the `default` case fails; a `_route:'background'` tool MUST have an explicit `case 'upload_file':` (mirror `case 'execute_js':` ~:374).
- **Shared chokepoint**: a single `background.js` helper `cdpSetFileInputFiles(tabId, selector, paths)` called by BOTH handlers — the denylist + audit live here so both front doors are covered.

## Element resolution
No existing CDP `DOM.*` consumer in the repo. Simplest path inside the helper: `DOM.getDocument{depth:0}` → root.nodeId → `DOM.querySelector{nodeId,selector}` → nodeId (0 = not found) → if not an `INPUT[type=file]` (check via `DOM.describeNode`), `DOM.querySelector` a descendant `input[type="file"]` (handles styled dropzones wrapping a hidden input) → `DOM.setFileInputFiles{nodeId, files}`. v1 requires a CSS `selector` + absolute `file_path` (background route skips content-side ref→selector resolution; ref support deferred).

## Security posture A (chokepoint in the helper, covers both front doors)
- **Denylist**: new `extension/utils/upload-path-denylist.js` (do NOT use the `capability-` prefix — that would force a recipe-path-guard allowlist entry). IIFE dual-export `globalThis.FsbUploadPathDenylist`; pure string policy; `importScripts` in background.js. Blocks `~/.ssh`, `~/.aws`, `~/.gnupg`, keychains, `.env*`, key/cert extensions, system secret stores, FSB vault.
- **Audit**: reuse `globalThis.FsbAuditLog.append({ts,origin,slug,method,sideEffectClass,consentDecision,outcome,error?})` (`extension/utils/audit-log.js`, Phase 30). The whitelist has NO `path` field by design, so persist origin + outcome + decision only — the full disk path is intentionally NOT persisted (avoids leaking filesystem structure into the audit store); the path stays in the transient `automationLogger` session log. This is a deliberate refinement of posture A.
- **Existence**: CDP `setFileInputFiles` errors if the file is missing/unreadable; the helper catches and returns a clean message. (MCP-side `fs.stat` pre-flight deferred — would need special-casing the generic execute-action path.)
