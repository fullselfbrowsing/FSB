# Plan 34-01 — Summary

Added `upload_file(selector, file_path, tab_id?)`: a first-class MCP + autopilot action tool that sets a real file from an absolute disk path onto an `<input type="file">` via CDP `DOM.setFileInputFiles` — the only mechanism that can populate a file input from disk (page JS is forbidden). Fills the file-upload "tool gap" the existing synthetic-only `drop_file` could not.

## Delivered

**The tool (UPLOAD-01)**
- `extension/background.js` `executeUploadFile`: `DOM.getDocument`→`querySelector`→`describeNode` (file-input check)→descendant `input[type=file]` fallback (handles styled dropzones)→`setFileInputFiles`; reuses the `chrome.debugger.attach('1.3')` stale-retry/detach seam; detach owned by `finally` so a detach hiccup never flips a real success to failure.
- `extension/ai/tool-definitions.js`: `upload_file` registry entry (`_route:'background'`, selector + absolute `file_path` required); `mcp/ai/tool-definitions.cjs` regenerated byte-identical.

**Two front doors, one chokepoint (UPLOAD-02)**
- MCP: `handleUploadFileRoute` + `MCP_PHASE199_TOOL_ROUTES.upload_file` (`mcp-tool-dispatcher.js`); tab ownership enforced by the existing gate (`TAB_NOT_OWNED`).
- Autopilot: `case 'upload_file':` in `executeBackgroundTool` (`tool-executor.js`).
- Both call the SAME `executeUploadFile`, so the denylist + audit cover both.

**Security posture A (UPLOAD-03)**
- `extension/utils/upload-path-denylist.js`: sensitive-path denylist (keys, credential stores, `.env*`, keychains, `/var/run/secrets`, `/proc/`, the FSB vault); absolute-path-only; Win32 trailing-dot/space stripped (closes a Windows bypass). Enforced BEFORE any side effect.
- Audit via `FsbAuditLog.append`: origin + outcome + decision only — the disk path is intentionally never persisted (avoids leaking filesystem structure); it stays in the transient session log.
