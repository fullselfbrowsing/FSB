## 1. System Classification

**System Type:** Autonomous Agent — delegated browser-control task agent

**Description:**
FSB delegates a side-panel task to the user's installed Claude Code process. A shell-free Node.js supervisor confines the CLI to FSB's loopback MCP server, streams provider output into a normalized event contract, and terminates the whole process tree on cancellation or drift. Good behavior means the correct installed binary is used, only FSB MCP tools are available, events remain observable and protocol-safe, and no user task, credential, or unrelated process crosses the boundary.

**Critical Failure Modes:**
1. Caller-controlled text or flags escape into a shell, argv, environment, inherited MCP configuration, or unrestricted built-in tools.
2. Cancellation or daemon failure leaves a Claude Code process or descendant controlling the browser.
3. Provider stream drift is silently accepted, misreported as success, or parsed differently by downstream UI code.
4. Detection fingerprints one executable/version but spawn runs another, including unsafe Windows `.cmd` fallback behavior.
5. Existing MCP wire contracts or Phase 59's authenticated reverse-channel guarantees regress.

## 1b. Domain Context

**Industry Vertical:** Local developer tooling / authenticated browser automation

**User Population:** FSB users who intentionally run the local `serve` daemon and a
user-installed, user-authenticated agent CLI; FSB maintainers and operators support the
integration.

**Stakes Level:** Critical for authority and credential containment; high for execution
reliability.

**Output Consequence:** Model decisions become FSB MCP tool calls against the user's real,
logged-in browser, while the daemon owns a local process tree. A false accept can mutate real
accounts or continue controlling the browser after cancellation; a false success can tell the
user work completed when it did not.

### What Domain Experts Evaluate Against

**Dimension: Delegation authority containment**  
**Good (domain expert would accept):** Only the explicitly started `serve` daemon can accept
an authenticated Phase 59 reverse request; the request cannot choose command, flags, cwd,
environment, config, or identity. The task is stdin-only, provider keys are scrubbed, inherited
MCP/tool surfaces are excluded, and Claude receives only the shipped FSB policy plus FSB MCP
tools.  
**Bad (domain expert would flag):** Any shell/string command, caller-supplied argv or config,
PATH re-resolution after detection, inherited user/project tools, or auto-approval surface.  
**Stakes:** Critical  
**Source:** ADAPT-02/03, CLAUDE-01/02; Phase 60 D-05..12 and D-18..23; Phase 59 trust boundary;
[Claude Code permissions](https://code.claude.com/docs/en/permissions).

**Dimension: Cancellation really revokes control**  
**Good (domain expert would accept):** Cancel settles exactly once only after the child and
matching descendants are gone; startup handles only journaled, strongly verified survivors
before advertising spawn capability; ambiguity fails closed without killing an unrelated
user-owned Claude process.  
**Bad (domain expert would flag):** Parent-only kill, unverified process-name scanning, silent
orphan adoption, cleanup before exit confirmation, or a cancel response while descendants can
still reach the browser.  
**Stakes:** Critical  
**Source:** ADAPT-04/05; Phase 60 D-13..17;
[Node.js child processes](https://nodejs.org/docs/latest-v22.x/api/child_process.html) and
[Microsoft `taskkill`](https://learn.microsoft.com/en-us/windows-server/administration/windows-commands/taskkill).

**Dimension: Browser ownership and vault boundaries survive delegation**  
**Good (domain expert would accept):** FSB mints the MCP agent identity; the delegated agent
acts only on its owned tabs, uses typed ownership recovery, and routes passwords, CVV, and saved
payment data only through vault-reference tools. Irreversible submissions remain human-confirmed
policy, even though the Phase 61 consent UI is not yet present.  
**Bad (domain expert would flag):** Caller-supplied `agent_id`, cross-agent tab access, needless
foregrounding/closing, credentials in prompts/tool args/logs, or automatic purchase, account,
permission, deletion, or public-post submission.  
**Stakes:** Critical  
**Source:** `skills/fsb/SKILL.md`, `multi-agent-contract.md`, and `vault-boundary.md`.

**Dimension: Outcomes and protocol state are truthful**  
**Good (domain expert would accept):** Spawn uses the exact fingerprinted binary/profile;
provider JSON ends at the adapter; bounded parsing drains stdout and stderr; every accepted event
normalizes to `{ type, sessionId, payload }`; malformed or unknown protocol input terminates as
`agent_protocol_drift` without replay or fabricated success; existing MCP bytes remain frozen.  
**Bad (domain expert would flag):** Silent event drops, raw Claude JSON above the adapter,
unbounded buffering/deadlock, weaker retry flags, automatic task replay, or a terminal success
without a validated result.  
**Stakes:** High  
**Source:** CLAUDE-03/04; Phase 60 D-24..28; Phase 59 final review and verification; PITFALLS 7/9.

### Known Failure Modes in This Domain

- **Control-plane injection:** untrusted task text gains shell, argv, environment, settings, MCP,
  or built-in-tool authority instead of remaining inert stdin data.
- **False cancellation or collateral cleanup:** the parent exits while descendants keep browser
  authority, or broad recovery matching kills an unrelated local CLI.
- **Compatibility drift disguised as progress:** a different binary/version or new JSONL shape is
  silently accepted, pipes stall, or the run reports success without a recognized result.
- **Browser-boundary collapse:** a delegated session steals another agent's tab, exposes a vault
  value, or performs a consent-requiring final action without a human handoff.

### Regulatory / Compliance Context

No sector-specific regulatory regime or certification is asserted for Phase 60: this is local
developer tooling, not an FSB-hosted AI service, payment processor, healthcare system, or
financial-advice product. Workloads may still contain regulated or personal data because the
agent acts in the user's authenticated browser. The applicable product controls are data
minimization and least privilege: FSB must not log/store the task in supervisor artifacts, proxy
OAuth tokens, expose provider API keys, or move passwords/PAN/CVV across the vault boundary.
These controls do **not** constitute PCI DSS, HIPAA, GDPR, SOC 2, or other certification.

Claude Code receives the task and permitted browser-tool results under the user's own account and
provider terms; FSB does not embed the Agent SDK or operate the model endpoint. Chrome consent UX,
service-worker persistence, and native messaging are Phase 61+ scope and cannot be claimed here.

## 2. Framework Decision

**Selected Framework:** No embedded AI framework — direct Claude Code CLI protocol integration behind `AgentProviderAdapter`

**Version:** Claude Code compatibility baseline `2.1.177`; Node.js built-ins on the package's supported Node `>=18.20.0`

**Rationale:**
Phase 60 integrates a user-owned, subscription-authenticated external runtime rather than hosting a model loop in-process. The direct CLI path preserves the user's existing login, keeps the security policy explicit in argv/stdin/MCP files, and lets the provider-neutral five-method adapter isolate Anthropic-specific details. LangChain/LangGraph would duplicate orchestration the CLI already owns, while the Claude Agent SDK conflicts with the locked no-embedded-SDK/subscription-auth boundary.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| Claude Agent SDK | Embeds a provider SDK/runtime, does not satisfy the locked installed-CLI subscription path, and would bypass the exact adapter/process contract. |
| LangGraph / LangChain | Adds runtime dependencies and an orchestration loop FSB neither needs nor owns in this phase. |
| PTY or terminal-screen scraping | Structured print-mode JSONL exists; terminal emulation expands attack surface and protocol drift. |

**Vendor Lock-In Accepted:** Partial and adapter-local. Claude invocation/fixtures are Anthropic-specific; supervision, normalized events, lifecycle handling, and the five-method contract remain provider-neutral.

## 3. Runtime Quick Reference

Phase 60 is a process/protocol integration, not an embedded model framework. The runtime is
the user's installed Claude Code executable, supervised with Node's built-in process and
stream APIs. FSB neither installs an SDK nor proxies authentication.

### Installation / Detection

- **External prerequisite:** Claude Code is user-installed and already authenticated. The
  official CLI supports `claude install 2.1.177`, but FSB must never run installation or
  login commands on the user's behalf.
- **Minimum gate:** resolve one executable path, execute that exact path with `--version`
  using `shell: false`, parse a semver-compatible value, and reject versions below
  `2.1.177`. Retain `{ realPath, version }` as one fingerprint; spawning may not perform a
  second PATH lookup.
- **Profile selection:** select a closed compatibility profile from the detected version.
  Record the exact version on each run so parser failures can name the profile and fixture.
  A newer version is not permission to weaken an unknown flag/event failure into a retry.
- **Windows:** accept a directly executable native binary. Node documents that `.cmd` and
  `.bat` files are not executable by themselves on Windows. A shim is acceptable only if a
  deterministic resolver verifies and returns its fixed native interpreter/entry point;
  otherwise detection reports unsupported. Never use `cmd.exe /c` or `shell: true`.
- **Help caveat:** the official CLI reference explicitly says `claude --help` does not list
  every flag. Therefore the documented `--max-turns` flag cannot be rejected solely because
  it is absent from the 2.1.177 help text; the closed profile and its automated argv tests are
  authoritative until live UAT at the milestone-end gate.

### Core Imports

```ts
import { spawn, type ChildProcessWithoutNullStreams } from 'node:child_process';
import { once } from 'node:events';
import { realpath } from 'node:fs/promises';
import { delimiter, extname, resolve } from 'node:path';
import { finished } from 'node:stream/promises';
import { z } from 'zod'; // already present in mcp/package.json; no new runtime package
```

Use asynchronous `spawn`, not `exec`/`execSync`: the command and every argument remain
separate values, output is streamed instead of buffered wholesale, and the daemon event loop
stays responsive.

### Entry Point Pattern

The adapter returns declarative spawn data; only the `serve`-owned supervisor creates the
process. Start all consumers before writing stdin so neither output pipe can fill while the
parent is waiting elsewhere.

```ts
async function runTask(task: string, ctx: SpawnContext): Promise<TerminalOutcome> {
  const adapter = registry.require('claude-code');
  const detected = await adapter.detect();
  const spec = await adapter.buildSpawn(task, { ...ctx, detected });

  // spec.command is the same retained real path that detect() fingerprinted.
  const child = spawn(spec.command, spec.argv, {
    cwd: spec.cwd,
    env: scrubProviderKeys({ ...process.env, ...spec.fixedEnv }),
    shell: false,
    detached: true,
    windowsHide: true,
    stdio: ['pipe', 'pipe', 'pipe'],
  });

  // Attach error/close observers and drain both pipes before supplying the task.
  const closed = once(child, 'close');
  const events = consumeEvents(adapter.parseEvents(child.stdout), ctx.emit);
  const diagnostics = drainBoundedAndRedacted(child.stderr, 64 * 1024);

  if (!child.stdin.write(task, 'utf8')) await once(child.stdin, 'drain');
  child.stdin.end();
  await finished(child.stdin);

  await Promise.all([events, diagnostics, closed]);
  return settleExactlyOnce(child, detected);
}
```

The real implementation must also register the child in the atomic orphan journal before it
can outlive the daemon, handle `spawn`/stdin errors, and route every failure through the same
exact-once terminal settlement path. The user's task is never interpolated into `spec.argv`,
`spec.fixedEnv`, a filename, a diagnostic, or the journal.

| Concept | What It Is | When You Use It |
|---------|------------|-----------------|
| `AgentProviderAdapter` | Provider-neutral contract with exactly `detect`, `buildSpawn`, `parseEvents`, `kill`, and `caps` | At every provider boundary; Claude-specific JSON and flags do not escape it |

| `SpawnSupervisor` | Sole owner of process creation, environment, stdin, lifecycle, journal, and event settlement | Only in the intentionally started `serve` daemon |
| Normalized `AgentEvent` | Strict `{ type, sessionId, payload }` event validated at runtime | Everything above `parseEvents`, including the reverse-channel harness and Phase 61 UI |

### Common Pitfalls

1. **Treating `--strict-mcp-config` as total isolation.** It excludes other MCP sources, but
   does not by itself disable project/user settings, hooks, skills, plugins, CLAUDE.md, or
   Chrome integration.
2. **Relying on a private `--settings` file alone.** The CLI reference says omitted keys keep
   their file-based values. A private file is an override, not an exclusion boundary.
3. **Using `--bare`.** Current docs say bare mode skips OAuth/keychain reads and still exposes
   Bash/read/edit built-ins. That violates both subscription authentication and the MCP-only
   surface.
4. **Assuming `system/init` is physically first.** Official headless docs allow plugin-install
   and hook events before init. In FSB's isolated profile these are configuration drift, not
   harmless noise; recognize them, fail the run, and do not fabricate an init/result.
5. **Waiting for exit before draining output.** Node pipes have limited platform-dependent
   capacity; an undrained stdout or stderr can block the child and deadlock cancellation.
6. **Calling `child.kill()` and declaring the task stopped.** POSIX descendants require group
   signaling and verification; Windows requires direct `taskkill /pid <pid> /T /F` plus
   verification. Microsoft documents `/T` as including children and `/F` as forceful.
7. **Launching a Windows command shim through a shell.** Node documents that `.cmd`/`.bat`
   need a terminal. Shell fallback reintroduces parsing and is forbidden; reject an unverified
   shim instead.
8. **Retrying after protocol drift.** Replaying an autonomous browser task can duplicate real
   effects. Unknown or malformed provider events terminate the one run; there is no weaker
   argv fallback and no automatic task replay.

## 4. Implementation Guidance

**Model Configuration:**
Do not pass `--model`, set provider credentials, invoke login, or inspect authentication by
inference. Claude Code owns the user's subscription/keychain authentication and model choice.
FSB only gates the executable at Claude Code `>=2.1.177`, scrubs `ANTHROPIC_API_KEY`,
`OPENAI_API_KEY`, and `GEMINI_API_KEY` from the child environment, and records the model value
reported by `system/init` as non-secret diagnostic metadata. No embedded Agent SDK or other AI
framework is installed.

**Core Pattern:**
Build one immutable argv profile and pass it as an array to `spawn`. The Phase 60 baseline is:

```ts
const args = [
  '-p',
  '--verbose',
  '--output-format', 'stream-json',
  '--include-partial-messages',
  '--setting-sources', '',          // must be proven by the 2.1.177 compatibility profile
  '--disable-slash-commands',
  '--no-chrome',
  '--strict-mcp-config',
  '--mcp-config', privateMcpConfigPath,
  '--agents', JSON.stringify({ fsb: shippedStaticAgent }),
  '--agent', 'fsb',
  '--permission-mode', 'dontAsk',
  '--tools', '',                    // disables built-ins; official docs say MCP is unaffected
  '--allowedTools', 'mcp__fsb',
  '--disallowedTools', 'Bash,Edit,Write,NotebookEdit,WebFetch,WebSearch',
  '--max-turns', '40',
  '--no-session-persistence',
] as const;

const child = spawn(detected.realPath, [...args], FIXED_SPAWN_OPTIONS);
// Only after consumers/listeners exist:
await writeWithBackpressureThenEnd(child.stdin, boundedTask);
```

`--agents` defines the shipped session-scoped `fsb` agent and `--agent fsb` makes that custom
agent the main thread. Official subagent docs explicitly support this composition and state
that the selected agent's system prompt, tool restrictions, and model settings govern the
whole session. The shipped JSON is static product policy; only the bounded user task enters
stdin. `--max-turns` is official even when an older `--help` rendering omits it.

**Customization isolation:** require the empty `--setting-sources` value to pass a controlled
2.1.177 compatibility check before the profile is enabled. Combine it with
`--disable-slash-commands`, `--no-chrome`, `--strict-mcp-config`, an owner-only MCP file, the
inline agent, `--tools ""`, and the explicit FSB permission rule. A private `--settings` file
may supply explicit values but cannot replace the empty source list because omitted settings
otherwise inherit. Do not put `--safe-mode` in the 2.1.177 profile merely because current docs
list it: the reference gives no introduction version, and it does not promise that explicit
`--agents`/`--mcp-config` remain active. Do not use `--bare`; it drops the required auth path.

At runtime, accept `system/init` only after validating that the built-in tool list is empty,
the only MCP server is `fsb`, and no plugins loaded. Understand documented pre-init
plugin/hook events well enough to report a typed configuration-drift failure; for this
profile, any such event is terminal because it proves isolation did not hold. Do not require
the `capabilities` array: official docs say it starts in 2.1.205 and is absent on 2.1.177.

**Tool Use:**
- Disable every built-in with the literal empty argument to `--tools`; keep the redundant
  built-in denylist as a source/argv tripwire.
- Load exactly one loopback MCP server from the private `--mcp-config` file and allow only the
  configured server. Claude permission rules accept `mcp__fsb` for all tools on server `fsb`
  (the explicit wildcard form is `mcp__fsb__*`). An unanchored `mcp__*` allow is invalid and
  must not be substituted.
- Use `dontAsk`, never `bypassPermissions`. Official docs say `dontAsk` denies calls that
  would prompt, while allowed MCP rules can run. Although read-only Bash can normally execute
  in this mode, `--tools ""` removes the Bash tool from model context.
- Treat a tool marked `anthropic/requiresUserInteraction` as denied in this unattended task
  mode. The current permission-mode docs document that behavior; the 2.1.177 profile must not
  claim later-version approval-card semantics that it cannot verify.

**State Management:**
Keep volatile active-run state in a supervisor-owned map keyed by the server-minted delegation
id. Persist only the minimum owner-only orphan identity needed for crash recovery: delegation
id, adapter/profile, PID/process identity, argv signature, and non-secret environment
fingerprint. Never persist task text, Claude output, tokens, auth data, or a resumable Claude
session. `--no-session-persistence` and capabilities `{ taskMode: true, chatMode: false,
resume: false }` make this explicit.

Start parser, stderr drain, exit/error observation, and cancellation observation concurrently.
All racing paths call one idempotent settlement primitive. On normal exit, cancellation,
stdin failure, protocol drift, or daemon shutdown, clean private runtime files and the journal
only after child exit and tree verification. Startup recovery examines only journaled
candidates and fails closed when identity is ambiguous.

**Context Window Strategy:**
FSB does not construct or replay a conversation and does not add RAG. One task produces one
non-persistent CLI session. The shipped persona contains durable policy; the user task is one
bounded stdin payload. Cap it well below Claude Code's documented 10 MB stdin ceiling (64 KiB
is the recommended Phase 60 limit), cap the run at 40 agentic turns, and let Claude Code own
its internal context compaction. Do not resume, summarize into a second run, or silently
truncate a task; reject an over-limit request before spawn with a typed, non-secret error.

## 4b. AI Systems Best Practices

The generic Pydantic prescription is intentionally replaced with strict TypeScript runtime
validation. The MCP package is TypeScript/ESM and already depends on Zod; adding Python,
Pydantic, a sidecar, or an embedded AI SDK would create a second runtime and a second schema
source without improving this JSONL process boundary.

### Structured Outputs with TypeScript Runtime Validation (Pydantic Equivalent)

Use strict schemas for FSB-owned input/output and discriminated, required-field schemas for
provider events. Preserve extra fields only inside a recognized Claude envelope so compatible
additions survive in `payload`; unknown discriminants and missing required fields fail loud.

```ts
import { z } from 'zod';

export const AgentEventSchema = z.object({
  type: z.enum([
    'init', 'assistant', 'assistant_delta', 'user',
    'tool_use', 'tool_result', 'retry', 'result', 'diagnostic',
  ]),
  sessionId: z.string().min(1),
  payload: z.record(z.unknown()),
}).strict();

export type AgentEvent = z.infer<typeof AgentEventSchema>;

// Provider objects may grow, but their required discriminants may not disappear.
const ClaudeBase = z.object({
  type: z.string().min(1),
  session_id: z.string().min(1),
}).passthrough();

export function validateNormalizedEvent(candidate: unknown): AgentEvent {
  const parsed = AgentEventSchema.safeParse(candidate);
  if (!parsed.success) {
    throw new AgentProtocolDrift('invalid normalized agent event', parsed.error.issues);
  }
  return parsed.data;
}
```

The production parser must define and test each known top-level type and required `system`
subtype rather than accepting `ClaudeBase` by itself. Frame JSON incrementally; recommend a
256 KiB per-line cap and a 64 KiB redacted stderr tail while allowing more than 200 KiB total
stdout across many lines. Validate the final unterminated line. Recursively `.strict()` the
FSB-owned `delegate.start`/`delegate.cancel` payloads so callers cannot smuggle command,
environment, path, or flags through nested objects.

There is **no protocol retry**. A schema/drift error emits one typed terminal diagnostic,
stops the child tree, and settles failed. Retrying or replaying the task could duplicate
browser mutations. Claude's documented `system/api_retry` is normalized as progress only; it
does not authorize an FSB-side replay. Log only error category, profile/version, event index,
and schema issue paths—never raw task, raw stdout, or credential-like values.

### Async-First Design

- Expose `parseEvents(stream)` as an async iterable and consume it as data arrives; do not
  buffer the whole stream or split each chunk as though chunks equal JSONL records.
- Attach `error` and `close` observers, stdout parsing, and bounded stderr draining before
  writing stdin. Honor `Writable.write()` backpressure, then `end()` stdin for one-shot print
  mode.
- Run parse, stderr drain, exit observation, cancel escalation, and reverse-channel emission
  concurrently. The common deadlock is awaiting child exit or one stream before draining the
  other; Node documents finite pipe capacity.
- Do not use synchronous child-process methods or call an event-loop bootstrap such as
  `asyncio.run()`—there is no Python runtime here. Use promises/async iterators within the
  daemon's existing event loop and one exact-once settlement function for all races.
- Stream normalized events for observability, but await validated terminal settlement before
  resolving `delegate.start`; await verified tree disappearance before resolving cancel.

### Prompt Engineering Discipline

- Keep system policy and user intent in separate channels: the static FSB persona is shipped
  and serialized into `--agents`; the untrusted task is bounded text written only to stdin.
- The persona must state the browser-control role, MCP-only tool boundary, ownership/vault
  rules, and fail-closed behavior for consent-requiring or irreversible actions. Because
  `--agent` replaces the default system prompt for the main session, include all FSB-specific
  safety and tool guidance there instead of assuming Claude Code's coding prompt remains.
- Never construct dynamic system-prompt JSON from task text and never add task-derived
  few-shot examples. Stable examples, if required, belong in the reviewed shipped persona;
  request-specific context belongs in stdin or in MCP tool results.
- Claude Code CLI does not expose a provider `max_tokens` parameter in this compatibility
  contract, so do not invent one or select a model. Bound work with the documented
  `--max-turns 40`, the task-size limit, process lifecycle limits, and MCP policy.

### Context Window Management

- Use one fresh session per task with `--no-session-persistence`; no prior transcript is
  injected and no resume id is exposed.
- Let the installed CLI own model-specific context compaction. FSB must not parse or rewrite
  provider history, because that would couple the adapter contract to Claude internals.
- Avoid echoing large MCP results back into prompts or normalized diagnostics. Tool schemas
  should return bounded, task-relevant observations; event consumers may persist UI-safe
  summaries later in Phase 61, never a hidden prompt transcript.
- At 40 turns, terminate with an explicit non-success outcome. Do not start a continuation
  run automatically; a second autonomous run could repeat side effects after losing context.

### Cost and Latency Budget

FSB cannot quote a reliable per-call token price: it deliberately does not choose the model,
billing path, or subscription tier. Its direct incremental cost is no embedded-model charge;
the user's Claude runtime owns any provider usage. If a terminal event includes usage/cost
metadata, preserve validated numeric fields for optional local observability but never make
success depend on them or infer billing from absence.

Use enforceable operational budgets instead: one child per delegation, at most 40 agentic
turns, a 64 KiB task, bounded individual event/stderr buffers, immediate incremental event
delivery, and deterministic cancellation grace/escalation. Record duration, time-to-init,
time-to-first-event, turns, tool calls, retries, bytes, and terminal reason without recording
content. Do not use exact-match or semantic response caching for stateful browser tasks—the
same text can have different effects as page/session state changes. Do not route subtasks to a
"cheaper" model; model choice remains user-owned and adapter-neutral.

## 5. Evaluation Strategy

Phase 60 is evaluated as a deterministic process/protocol boundary. Every critical security,
lifecycle, and wire-contract case is binary and must pass; percentages cannot average away a
single authority leak, orphan, or fabricated result. The existing Node test harness imports the
compiled MCP output, injects fake child/process/platform dependencies, and replays a sanitized
Claude Code 2.1.177 JSONL recording plus deterministic mutations. No live Claude login, model
call, browser action, or real OS process-tree claim is part of this phase's automated result.

### Dimensions

| Dimension | Rubric (Pass/Fail or 1-5) | Measurement Approach | Priority |
|-----------|--------------------------|----------------------|----------|
| **Safety — delegation authority containment** | **PASS:** recursively strict `delegate.start`/`delegate.cancel` validation rejects every caller-controlled command/argv/env/cwd/config/id field; the bounded task appears only in the exact bytes written to stdin; provider keys are absent from the child environment, journal, and diagnostics. **FAIL:** any untrusted value reaches a control plane, inherited surface, shell, or persistent artifact. | Code: fake-spawn capture, hostile-input/canary cases, strict-schema assertions, and source/argv tripwires | Critical |
| **Tool-use and browser-policy containment** | **PASS:** the fixed profile disables built-ins, exposes only `mcp__fsb`, selects the static shipped `fsb` agent, and the agent policy states ownership, vault-reference, and irreversible-action handoff rules; validated init attests to that surface. **FAIL:** any built-in, extra MCP server/plugin, caller-defined agent, raw credential path, or auto-approval surface is available. | Code: exact argv/config/persona assertions and synthetic init attestation; real browser-policy behavior is Human/live and deferred | Critical |
| **Cancellation and crash-recovery safety** | **PASS:** fake POSIX/Windows state machines prove TERM/grace/KILL or direct `taskkill` ordering, exact-once settlement only after exit plus descendant verification, journal-only recovery, and fail-closed ambiguity without collateral kill. **FAIL:** parent-only cancellation, early resolution, double settlement, broad name matching, or acceptance of new work with an unresolved verified orphan. | Code: injected fake child, clock, process inspector, taskkill, and journal helpers; real OS behavior remains deferred UAT | Critical |
| **Protocol truthfulness and output validity** | **PASS:** the pinned fixture and valid mutations normalize only recognized events to strict `{ type, sessionId, payload }`, preserve compatible fields, handle chunk boundaries/final lines, and produce one validated terminal result. Unknown/malformed shapes become `agent_protocol_drift`, kill the run, and never fabricate success. **FAIL:** silent drop, raw Claude JSON above the adapter, retry/replay, or a success without a recognized result. | Code: recorded-fixture replay, generated mutations, strict runtime schemas, and terminal-event assertions | Critical |
| **Task completion and terminal integrity** | **PASS:** a valid offline run emits the expected ordered progress/result labels, drains both pipes, observes child close, resolves the long-lived start correlation once, and treats non-result exit, stdin error, cancel, or drift as explicit non-success without replay. **FAIL:** premature success, duplicate final response, hung correlation, or automatic second attempt. | Code: reverse-channel harness plus fake child/error/race matrix; semantic browser-task quality is Human/live and deferred | Critical |
| **Compatibility and binary identity** | **PASS:** detection binds one real path and parsed version to spawn, admits the closed 2.1.177+ profile, rejects old/unparseable/changed binaries and unsafe Windows shims, and reports a typed diagnostic. **FAIL:** PATH is resolved twice, help-text absence weakens the profile, or a shell/string fallback runs. | Code: injected resolver/version probe/platform matrix and exact spawn-capture assertions | Critical |
| **Resource and turn-budget adherence** | **PASS:** tasks over 64 KiB and JSONL lines over 256 KiB reject before unsafe buffering, stderr retains at most a redacted 64 KiB tail, more than 200 KiB total stdout drains incrementally, and the profile fixes 40 turns. **FAIL:** truncation presented as success, unbounded accumulation, blocked pipe, missing limit, or budget bypass. | Code: generated boundary payloads, backpressure-aware fake streams, byte counters, and argv assertion | High |
| **Existing wire and package invariants** | **PASS:** Phase 59 authentication/routing remains authoritative, spawn capability exists only in `serve`, the five legacy transport error codes and frozen serialized frames remain exact, and the package includes the static agent/fixture assets. **FAIL:** stdio/incidental hubs can spawn, legacy bytes change, auth weakens, or required assets are absent. | Code: reverse-channel/real-socket harness, byte-freeze/parity tests, source-shape checks, and package-content assertions | Critical |

No LLM judge is used. These pass/fail criteria concern exact bytes, schemas, argv, process state,
and lifecycle ordering, for which an LLM judge would be less reliable than code. Whether Claude
understands a real browser task and chooses good actions is a separate human/live dimension; it
stays pending for the single milestone-end UAT gate rather than being simulated or claimed here.

### Eval Tooling

**Primary Tool:** The repository's existing plain Node.js test runner against
`mcp/build/**`, with `node:assert/strict`, injected fake process/stream/platform helpers, checked-in
fixtures, the TypeScript build, and existing source/byte tripwires. No new eval framework or
service is required.

**Setup:**

```bash
npm install
npm --prefix mcp install
npm --prefix mcp run build
```

The first two commands are normal repository setup, not an eval-specific dependency install.
The implementation should add focused built-output tests for the adapter contract, Claude
profile/parser, supervisor, and orphan recovery, then include them in the root serial test chain.

**Arize Phoenix override:** no tracing/eval package was found in the production or test
manifests, and Phoenix is intentionally not introduced. It would add a Python/collector or
service dependency to a TypeScript local CLI/protocol integration, create a tempting export path
for prompts, provider events, and browser tool results, and provide little value for assertions
that are deterministic at the byte/schema/process level. Secret-free local counters and bounded
typed diagnostics cover Phase 60 observability. Promptfoo and an LLM judge are likewise
unnecessary because this phase does not compare prompt/model variants or score subjective text.

**CI/CD Integration:**

The focused gate should run before the full repository regression chain (filenames may be grouped

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| Authenticated, `serve`-only spawn surface | Request arrives outside the Phase 59 authenticated local-capable route, before recovery completes, or in stdio/incidental-hub mode | Reject before adapter lookup; do not advertise `agent-spawn` |
| Closed request schema and bounded stdin | Unknown/nested control key, non-canonical adapter id, invalid delegation id, invalid UTF-8, or task over 64 KiB | Reject before detect/spawn with typed, non-secret error; never truncate |
| Fingerprint/profile/config containment | Binary identity/version changes, profile invariant fails, unsafe Windows shim appears, provider key survives scrubbing, or private runtime file cannot be secured | Do not spawn; clean safe temporary state; emit category/profile diagnostic without values |
| Init tool-surface attestation | Init reports any built-in tool, non-`fsb` MCP server, loaded plugin, missing FSB server, or mismatched session | Kill the affected tree, settle failed, and emit typed configuration drift before accepting browser-ready state |
| Bounded strict event parser | Line exceeds 256 KiB, JSON/schema/discriminant is invalid, event order is impossible, or a terminal result is missing | Emit one `agent_protocol_drift` diagnostic, kill, settle failed, and never replay |
| Exact-once lifecycle and tree verification | Cancel/shutdown/error/drift races, grace expires, descendants remain, or journal recovery is ambiguous | Idempotent settlement; TERM→KILL/`taskkill` as applicable; wait for verified absence; withhold spawn capability on unresolved recovery |
| Secret-free bounded diagnostics | Task/provider-key/secret canary reaches a sink, stderr exceeds its tail budget, or raw provider content is offered to monitoring | Drop/redact content, retain only typed counters/paths, and fail the applicable test/run closed |

### Offline (Flywheel)

| Metric | Sampling Strategy | Action on Degradation |
|--------|-------------------|-----------------------|
| 20-case reference-label pass rate | 100% on every relevant change and every CI run | Any critical failure blocks merge; add a minimized regression case before fixing |
| Fixture mutation coverage | Replay all known types plus chunk, missing-field, unknown-type, oversized-line, and final-line mutations on every parser change | Extend manifest/parser profile; never broaden to accept unknown events silently |
| Spawn-policy/source-tripwire coverage | 100% exact argv/env/stdin/config assertions and forbidden-string scan on every agent-provider change | Block merge; review profile and static agent as security-sensitive code |
| Lifecycle state-transition coverage | Full injected ordering matrix for spawn error, close, result, cancel, escalation, shutdown, and recovery on every supervisor change | Block merge; add the newly observed interleaving to the matrix |
| Phase 59 byte/parity regression | Existing frozen frames, five error codes, real-socket topology, and version parity on every root test run | Block merge and preserve additive integration; do not update snapshots to hide drift |
| Secret-canary leakage rate | 100% of hostile reference cases plus existing redaction/log tests | Required value is zero; any hit blocks merge and triggers review of all sinks/artifacts |
| Real CLI/OS/browser behavior | 100% of designated live cases once, at the user-directed milestone-end gate only | Keep status pending until a human executes it; record failures without inventing a Phase 60 pass or replaying browser effects |

## 7. Production Monitoring

**Tracing Tool:** No external trace collector. Use the existing bounded local diagnostic ring and
supervisor-owned in-memory counters/timers with a closed, secret-free schema. Phoenix,
OpenTelemetry export, raw transcript capture, and provider-event persistence are outside this
phase. Phase 61 may render normalized UI-safe events, but it must not turn them into prompt/tool
result telemetry.

**Key Metrics to Track:**

- Starts accepted/rejected by stable reason category, adapter profile/version, and terminal
  outcome (`success`, `cancelled`, `spawn_failed`, `protocol_drift`, `tree_unsettled`,
  `daemon_restart_lost_run`), without task text or stable user identity.
- Time to detect, spawn, validated init, first normalized event, terminal event, child close, and
  verified cancellation; grace escalation and still-matching-descendant counts.
- Normalized event/turn/tool-call/retry counts and stdout/stderr byte counts only; never event
  payloads, tool arguments/results, raw stderr, or model reasoning.
- Schema/configuration drift counts by profile, provider top-level type/subtype category, event
  index, and schema issue path; do not retain the rejected line.
- Init attestation failures (unexpected built-in/MCP/plugin), oversize task/line rejections,
  bounded-stderr truncations, and exact-once settlement invariant violations.
- Orphan journal records examined and outcomes (`confirmed_killed`, `stale_cleared`,
  `ambiguous_fail_closed`), plus whether spawn capability remained withheld.
- Redaction-canary detections and provider-key removals as counts only. The accepted value for raw
  secret/task content in any log or artifact is always zero.

**Alert Thresholds:**

| Signal | Threshold | Required response |
|--------|-----------|-------------------|
| Authority/tool-surface violation, raw secret/task leak, exact-once violation, collateral-process candidate, or unresolved matching descendant | Any single occurrence | Immediate critical diagnostic; stop/withhold affected spawn capability and require maintainer review |
| `agent_protocol_drift`, missing terminal result, or unsupported profile at runtime | Any single run | Fail that run without replay; surface adapter/profile diagnostic and require fixture/profile review before compatibility is changed |
| Cancellation latency | Any unresolved tree after grace plus inspection, or p95 greater than configured grace + 2 s over at least 20 local cancels | Withhold new spawn on unresolved tree; investigate platform helper before release |
| Init attestation failure | Any unexpected built-in/server/plugin; or more than 1 failure in 20 starts for non-drift operational errors | Kill before browser-ready state; review hermetic profile/configuration |
| Buffer bounds | Any line over 256 KiB or stderr over 64 KiB tail | Apply the guardrail; warn by typed counter only; never include discarded content |
| Detection regression | 3 consecutive failures for a previously supported retained path/version | Mark adapter unavailable and direct the user to secret-free doctor diagnostics |
| CI/reference regressions | Any critical case below 100%, any frozen-byte change, or any secret-canary hit | Block merge; no retries or snapshot refresh without reviewed contract change |

**Smart Sampling Strategy:**

Every run executes online guardrails and emits only cheap content-free counters. Successful runs
retain aggregate timings/counts; failed runs retain one bounded typed diagnostic with category,
profile, event index, and schema paths, never payload content. Offline CI replays 100% of the
20-case dataset rather than sampling it. Weight any later manual review toward drift, retry,
escalation, orphan, long-duration, and buffer-limit signals, but review the live browser/task only
at the milestone-end UAT gate. Do not automatically replay a sampled task and do not upload local
diagnostics to a hosted service.

Phase 60 defines and tests this monitoring schema but has no production delegation UI or deployed
telemetry pipeline. Baselines from real Claude authentication, OS process trees, crash recovery,
and browser work remain explicitly pending until milestone-end UAT.
