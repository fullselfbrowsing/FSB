<interfaces>
The journal schema is version 1 with exact top-level keys `version` and `entries`. Entries form a strict two-state union. A durable `prepared` entry has only `state`, `delegationId`, `adapterId`, `profileVersion`, `createdAt`, `binaryRealPath`, `argvSignature`, and unique non-secret `envFingerprint`. An `active` entry has those keys plus `pid`, `processGroupId`, `startedAt`, and `processStartIdentity`. No prompt/task/output/session transcript/credential field is permitted. Prepared state is written before spawn; active state is written before task stdin/browser authority is released.

The inspector returns one of `confirmed`, `stale`, or `ambiguous`, with descendants only for a confirmed matching identity. PID existence or executable name alone is never confirmation.
</interfaces>

Define injectable `ProcessInspector`, signal/spawn, child-exit, monotonic-clock, and timer dependencies. A production inspection request is always anchored by one journaled active PID/group or one unique prepared fingerprint and must compare process start identity, exact fixed argv signature, and the non-secret FSB environment fingerprint before returning `confirmed`; missing evidence is `stale` only when absence is proven, otherwise `ambiguous`. Never enumerate/kill by process name, broad `pkill`, or PID alone.

Implement concrete bounded backends rather than an unspecified native facility:

- **Linux:** read only numeric `/proc` entries using `node:fs`; parse `/proc/<pid>/stat` for PID/PPID/PGRP/start ticks, `/proc/<pid>/cmdline` for NUL-delimited exact argv, and `/proc/<pid>/environ` for the exact FSB fingerprint. Descendants are derived from PPID links. Permission, parse, truncation, PID-reuse, or missing required file is ambiguous unless all relevant paths prove the process absent.
- **Darwin:** call the absolute native `/bin/ps` (or verified absolute platform path) with fixed argv and `shell:false`. Use a bounded PID/PPID/PGID/start/command table for tree relationships, then a bounded `-E -ww -p <pid>` query for the exact journaled candidate's argv/environment evidence. Never log the output. Nonzero exit, truncation, ambiguous parsing, missing exact flag sequence/fingerprint/start identity, or multiple prepared matches is ambiguous.
- **Windows:** resolve only the native `%SystemRoot%\\System32\\wbem\\wmic.exe` regular executable and call it with fixed argv `['process', 'get', 'ProcessId,ParentProcessId,CreationDate,CommandLine', '/format:csv']`, `shell:false`, and bounded output. If WMIC is absent/disabled, parsing is partial, or the zero-dependency result cannot prove the exact FSB environment fingerprint, return `ambiguous`, withhold spawn, and send no kill. Do not use PowerShell/cmd string scripts, WMI name-only matching, or downgrade to PID/command-line-only confirmation.
- **Other/unsupported:** always return `ambiguous`; no signal is sent and spawn remains unavailable until an operator clears the safely diagnosed journal condition.

Cap every inspection read/output and process count; partial/truncated evidence is ambiguous. Tests invoke each production-path parser with filesystem/exec fixtures, not only a generic fake, and prove unavailable/partial evidence never signals.

Implement POSIX stop as `process.kill(-processGroupId, 'SIGTERM')`, wait concurrently for child exit and inspector tree state through the configured grace, then `process.kill(-processGroupId, 'SIGKILL')` if a confirmed matching tree remains. ESRCH/missing becomes stale only after inspection; permission/identity uncertainty is not success. Implement Windows stop by spawning the retained native `taskkill` executable directly with `['/pid', String(pid), '/T', '/F']`, `shell:false`, `windowsHide:true`, bounded drained stdio, then inspect. Do not call `child.kill()` as the complete solution and do not use `cmd /c`.

Return only after child `close`/spawn-error settlement plus verified absence/stale classification. If descendants remain or inspection is ambiguous, return a typed `tree_unsettled` failure. Make duplicate calls/racing exit idempotent at the helper boundary. Tests use fakes to enumerate TERM-no-descendants, TERM-then-KILL, already-stale, signal errors, Windows exact argv, taskkill failure, delayed close, remaining descendants, ambiguity, duplicate stop, and unrelated-process cases.
  </action>
