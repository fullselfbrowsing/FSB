<interfaces>
The authenticated daemon response is the browser-safe projection established in Plan 62-01, with no local-only doctor fields:

```ts
import type { CompatibilityReason } from './compatibility.js';

type AdapterCompatibilityResponse = Readonly<{
  schemaVersion: 1;
  checkedAt: number;
  adapters: readonly Readonly<{
    adapterId: string;
    displayLabel: string;
    status: 'supported' | 'degraded' | 'unsupported';
    reason: CompatibilityReason;
  }>[];
}>;
```

`CompatibilityReason` is imported from the canonical Plan 62-01 module on the daemon side. The browser validator accepts exactly `within_tested_range`, `newer_than_tested_range`, `evidence_stale`, `binary_not_found`, `version_missing`, `version_malformed`, `below_minimum`, `wrong_major`, `adapter_unshipped`, or `matrix_invalid`; it rejects every other string. The bridge wrapper sends exactly method `adapter.compatibility` with exactly `{}`. The provider envelope gains one dedicated compatibility submap/snapshot while preserving existing `clicked`, `connected`, `installed`, and unknown forward-compatible keys. A merged agent row exposes compatibility separately from installation, connection, recommendation, auth, and billing evidence.
</interfaces>

<threat_model>
Boundary: authenticated daemon facts cross the reverse channel into extension background memory, durable browser storage, and UI-facing merged rows.

- Authority: the daemon registry/detectors and `compatibility.ts` alone classify versions. The bridge transports; background validates, applies freshness, writes, and fans out; UI consumers may not inspect binaries or compare versions.
- Exact keys and bounds: request payload is an own plain object with zero keys. Response top-level keys are exactly `schemaVersion`, `checkedAt`, and `adapters`; rows are exactly `adapterId`, `displayLabel`, `status`, and `reason`. Require schema 1, finite non-negative safe-integer timestamps, at most 16 rows, unique IDs, strings at most 64 characters, and the closed status/reason vocabularies from Plan 62-01.
- Malformed/prototype failure: reject sparse arrays, inherited/unknown keys, accessors, poisoned prototypes, duplicate IDs, non-finite timestamps, and partial rows before storage. Malformed/timeout/auth/offline results preserve only previously validated evidence and project stale/unsupported; they never assert fresh support.
- Forbidden data: request, response, storage, logs, and fan-out must never include bridge secrets, auth payloads, sessions, provider output, prompts, tasks, environment values, binary paths, detected versions, profile internals, or local doctor metadata.
- Idempotence/concurrency: coalesce concurrent refreshes; serialize replacement on the existing mutation chain; await durable write before success/fan-out; repeated identical snapshots do not mutate selection/recommendation/provider settings. A rejected write cannot leak a newer in-memory supported result.
- Negative tests: unauthenticated request, non-empty payload, wrong schema, unknown/accessor/prototype keys, oversized/duplicate/sparse rows, bad clock, timeout/disconnect, storage rejection, corrupt cache, stale cache, concurrent refresh, and reconnect replay.
- Security gate: unresolved High/Critical auth bypass, secret/path/version leak, fail-open support, stale-as-supported projection, lost envelope data, or write-before-validation bug blocks completion.
</threat_model>
