<decisions>
## Implementation Decisions (locked by the spike + the user's full-parity choice)

### Fetch + vendor
- **Mechanism:** `gh api repos/opentabs-dev/opentabs/tarball/<SHA>` (one ~49MB hermetic download),
  extract, copy each non-fixture plugin's `package.json` + `src/**` (index.ts, `<app>-api.ts`,
  tools/*.ts incl schemas.ts) into `vendor/opentabs-snapshot/plugins/<app>/`. Real `src` — NO
  sdk-stub / `<app>-api` stub (the real `@opentabs-dev/plugin-sdk` imports cleanly under tsx).
  Exclude the 2 CI fixtures (`e2e-test`, `prescript-test`) via the existing FIXTURE_DIR_RE. Net 117.
- **SDK:** no stub needed; keep `@opentabs-dev/plugin-sdk@0.0.113` + `@opentabs-dev/shared` +
  `zod@4.4.3` + `tsx@4.22.4` as devDependencies (installed). Wall 1 holds (build-time only, no
  `handle()` executed, pure-data descriptors shipped).
- **SHA:** 4b17021637d2cac12b8d84d21c40e765aa7b85e9 (MIT). Update vendor PIN.md + _provenance.json.

### AUGMENT, not replace (critical)
- Import the 117 real apps (minus grafana/sqlpad empty-origin; carving target.apply_promo_code) →
  ~2,372 ops. The 40 overlapping hand-authored apps get their descriptors REGENERATED from real
  source (the real op sets are the source of truth — richer/more accurate). 77 real apps are net-new.
- **PRESERVE the 13 hand-authored-only apps** (NO upstream at the SHA): amazon, etsy, eventbrite,
  grubhub, kayak, lyft, mastodon, opentable, shopify, stubhub, threads, ticketmaster, ubereats.
  Keep their existing descriptors + classifications; mark "hand-authored, no upstream@SHA" in
  provenance. (These are payment/commerce + mastodon that 38/39 deliberately screened SENSITIVE.)
- Net corpus: ~117 real + 13 hand-only ≈ 130 apps, ~2,400+ descriptors. EXISTING_HEAD_SERVICES
  (github/slack/notion) still skipped (heads not clobbered).

### Stem collisions (BLOCKER A — mechanical)
- Extend STEM_OVERRIDES with a canonical stem for all ~40 wrong/colliding-stem apps. Disambiguate
  the 6 collision groups: `console` (aws-console→aws, clickhouse→clickhouse, google-cloud→gcloud,
  twilio→twilio), `www` (google-maps/npm/reddit/yelp), `atlassian` (jira/confluence — already),
  `cloud` (mongodb-atlas/temporal-cloud), `slack` (slack/slack-enterprise→slackent), `web`
  (telegram/whatsapp). Plus ~28 single wrong stems (azure→azure, bluesky→bsky, cloudflare→cloudflare,
  datadog→datadog, stripe→stripe, spotify→spotify, etc.).
- **Add a CI no-duplicate-stem assertion** (a full import MUST ship the complete override map or
  collisions silently clobber descriptors).

### Carves / skips (BLOCKER B, C — enumerated)
- `target.apply_promo_code` trips the Wall-1 forbidden-field pre-scan (input field `code`, the ONLY
  forbidden-in-input op corpus-wide). **Decision: a narrow, audited per-op input-field rename
  `code`→`promo_code` for this single business-`code` op** — preserves the op + keeps `script/expr/
  transform/fn/js` ALWAYS-fatal (the eval-able names). (Fallback if the rename adds risk: skip the
  1 op. Claude's discretion at execute time, defaulting to the rename.)
- `grafana` + `sqlpad`: empty `urlPatterns` (self-hosted, runtime instanceUrl). **SKIP both real
  apps** for this pass (no static origin → don't fit the origin-keyed denylist/search model). The
  hand-authored `grafana` slice (grafana.com) is PRESERVED (augment), so grafana stays covered;
  sqlpad (no hand-authored) is dropped (1 app). Self-hosted handling is a separate future story.

### Re-screening (the security-critical task)
- The fail-closed `classifyGate` runs merge-time per batch (already does). It surfaces exactly **2**
  origins needing a decision today: **linkedin.com** (social axis) + **youtube.com** (media axis) —
  classify both (linkedin→sensitive, youtube→denied/sensitive per the ToS-axis) before their batch
  lands.
- **Commerce/payment reconciliation — CONSERVATIVE (the established posture):** keep commerce/
  payment/travel brand origins **SENSITIVE** even though their REAL ops are largely read-only at
  this SHA (real doordash/booking/walmart = list/get, no place_order). Rationale: a sensitive origin
  runs READS under Auto (no friction) but GATES any WRITE — so keeping them sensitive is fail-safe
  against a future op-set change without hurting today's read-only UX. The payment-op CI guard
  (covers all payment verbs/op-names) + the per-op sideEffectClass enforce that NO payment op is
  ever safe-and-API-invocable. Do NOT downgrade a commerce/payment brand to safe just because its
  current real ops are read-only. (READ_ONLY_SAFE is only for genuinely-non-commerce read origins.)

### Scale (the headline perf risk)
- Index projects to ~1.31MB at 2,374 ops (579.9 B/entry) — inside the SCALE-01 ~1-2MB band but upper
  half. MEASURE the real serialized index on the full corpus and ASSERT `< ~2MB` + `loadJSON+first-
  search < ~100ms` in the eval harness. Keep the synthSynonyms cap (6/op); trim `description` from
  storeFields or shard ONLY if it overshoots. (Phase 43 owns the authoritative SCALE-01 + milestone
  gate; this phase proves the breadth corpus is within budget.)

### Invariants
- NO change to the importer core path. Wall 1 (no opentabs/sdk runtime shipped; recipe-path-guard
  green). INV-01 (recipe-index.generated.js IIFE/djb2 shape; only DATA grows). The HEAD_HANDLER cap.
  Per-wave/per-batch snapshot regen + the screen-then-import contract (Phase-36 blocker class).

### Claude's Discretion
- Exact category-batching of the 117 apps (reuse enumerateBatchApps; batch by category for
  reviewable screening, or fewer larger batches — the dry-run imported all 117 in one pass).
- The exact STEM_OVERRIDES canonical names (avoid new collisions; the no-dup-stem CI gate enforces).
- target.apply_promo_code rename-vs-skip (default rename).
- Whether to trim storeFields if the index overshoots 2MB.
