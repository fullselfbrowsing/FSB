# Phase 39.5 (Full OpenTabs Source Import) -- Security-Logic Review

**Reviewed:** 2026-06-26
**Depth:** deep (cross-file + empirical: ran every gate, classified all 130 emitted hosts, traced all 2,383 descriptors against the vendored source)
**Status:** issues_found (2 HIGH, 2 MEDIUM, 1 LOW)

## Summary

The headline security mechanics of Phase 39.5 are CORRECT and were verified empirically against the real corpus:

- **Apex-origin bug fix (commerce):** VERIFIED. Every one of the 32 commerce/payment/travel/marketplace brands classifies SENSITIVE on EVERY emitted host form (apex AND www). 0 commerce brands ship safe-under-Auto. The `*.<brand>` apex-suffix widening in `service-denylist.json` correctly suffix-matches both `brand.com` and `www.brand.com` (confirmed in `service-denylist.js _matchesPattern`). All 26 payment ops (place_order/checkout/book_*/buy_*/request_ride/...) are GATED, backing:'dom', 0 safe-and-invocable.
- **Commerce backstop (`checkCommerceSensitiveClassified`):** VERIFIED build-failing, non-bypassable, fail-closed. Exercised it directly: a forced-safe brand -> 1 failure; null classifyFn -> 32/32 failures (fail-closed); disjointness with READ_ONLY_SAFE asserted and holds.
- **classifyGate DEF-39.5-03-A fix (drop op-prose from the screen):** VERIFIED it does NOT weaken the screen. All 20 SAFE origins whose description-only would trip an axis are genuinely-benign dev/infra tools (circleci "health", cockroachdb "invoices", glama "claude", zillow "tax", etc.). NONE is a real finance/health/social brand. No genuinely-sensitive origin slips because its sensitivity lived only in prose. Host + slug still screen every origin; payment-verb op-names still trip; orthogonal to the Wall-1 input pre-scan (which is input-schema-only and unchanged).
- **promo_code carve:** VERIFIED narrow. `target.apply_promo_code` emits `promo_code` (not `code`); the FORBIDDEN set is untouched; 0 forbidden fields (script/expr/transform/code/fn/js) across all 2,383 descriptor params.
- **no-duplicate-stem gate:** VERIFIED via the importer's own exports: 127 enumerated apps -> 127 distinct stems, 0 collisions; all 2,383 opentabs slugs unique.
- **Wall 1:** VERIFIED. No opentabs/zod/SDK runtime in `extension/`; `vendor/opentabs-snapshot` is not referenced by `manifest.json`/`dist/`.
- **Posture-B writes:** VERIFIED. reddit (www.reddit.com), calendly (calendly.com), discord all classify sensitive; their write/destructive ops are gated.

HOWEVER, the augment ("preserve the 13 hand apps") was NOT clean. The full-source import ADDED the real apex-host descriptors but the importer has **no prune step**, so it left **94 STALE (orphan) descriptors** from the earlier hand-authored Phase-38/39 slices on disk -- referencing source `.ts` files that no longer exist in the vendored snapshot. This produces (a) a genuine break in the READ_ONLY_SAFE invariant for `tripadvisor.com`, and (b) broad catalog/seed/provenance corruption. All gates are GREEN, so these escape CI silently.

## High

### HI-01: READ_ONLY_SAFE invariant is broken for `tripadvisor.com` (apex) -- a write op ships safe-under-Auto, and the crosscheck FALSELY reports "tripadvisor emits read-only ops"

**STATUS: FIXED.** `tripadvisor.check_saved` reclassified `read` (verified GraphQL query mislabel) via a narrow per-op `SIDE_EFFECT_READ_CONFIRMED` allowlist in `scripts/lib/side-effect-class.mjs` (keyed by full slug; cannot downgrade a real write/destructive signal -- meticulous.check_for_flakes stays write). `tripadvisor.com` apex added to `READ_ONLY_SAFE_SERVICES`, and `checkReadOnlySafeOrigins` made host-form-agnostic (www<->apex). After the HI-02 prune the 3 stale www.tripadvisor.com orphans are gone, leaving the apex ops; the read-only invariant now genuinely covers the emitted origin and a future tripadvisor write FAILS the build. `inferSideEffect` now passes the full `<stem>.<op>` slug so the importer stamp == the gate derivation.

**File:** `scripts/verify-catalog-crosscheck.mjs:210-213` (READ_ONLY_SAFE_SERVICES) + `scripts/verify-catalog-crosscheck.mjs:225-247` (checkReadOnlySafeOrigins)

**Issue:** This is the EXACT apex-vs-www bug class the phase set out to close -- closed for commerce (via `*.` patterns) but MISSED for the read-only-safe roster (which uses EXACT host strings).

`READ_ONLY_SAFE_SERVICES` lists only `www.tripadvisor.com` (and `www.yelp.com`, `www.zillow.com`+`zillow.com`, `grafana.com`). But the live vendored tripadvisor slice has `urlPatterns: ["*://*.tripadvisor.com/*"]`, so `readPluginMeta` strips `*.` and emits all 12 current ops under the APEX `tripadvisor.com`. One of them is a write:

```
tripadvisor.check_saved   service=tripadvisor.com   class=write   backing=dom
  signals: { transportHelper:"graphql", httpMethod:null, opNameVerb:"check" }
```

`tripadvisor.com` (apex) is NOT in READ_ONLY_SAFE_SERVICES and is NOT in the denylist, so it classifies SAFE. Therefore:
1. `checkReadOnlySafeOrigins` only inspects the `www.tripadvisor.com` descriptors (the 3 stale ones, all read) -> passes. The apex form -- including the write -- is never checked.
2. The crosscheck CLI prints "every read-only-safe origin (yelp/**tripadvisor**/zillow/grafana) emits read-only ops, MED-02" -- this success message is FALSE.
3. `check_saved` (write, backing:dom) on a SAFE origin runs under Auto with no mutating re-gate.

Today's real-world payload is low: `check_saved` is semantically a read mislabeled `write` (GraphQL ambiguous-verb fail-safe-high). But the **safety invariant is genuinely broken**, and the escalation path is live: if the tripadvisor slice is ever re-vendored with a real write (e.g., `save_location`), it ships writable-under-Auto AND escapes the build-failure that is supposed to force a re-classification decision. Note `zillow` was correctly added with BOTH `zillow.com` and `www.zillow.com` in the same 39.5-04 commit -- tripadvisor's apex was simply overlooked.

**Fix:** Add the apex form to the read-only-safe set so the invariant covers the EMITTED host (matching the zillow precedent):

```js
const READ_ONLY_SAFE_SERVICES = new Set([
  'www.yelp.com',
  'www.tripadvisor.com', 'tripadvisor.com',   // <-- apex emitted; cover it
  'www.zillow.com', 'zillow.com', 'grafana.com',
]);
```

Better: make `checkReadOnlySafeOrigins` membership host-form-agnostic (strip a leading `www.` before the set lookup), so a brand listed in either form covers both -- the same apex/www robustness the commerce path gets for free from `*.` patterns. Either fix will immediately turn `check_saved` into a build FAILURE, forcing an explicit decision (reclassify tripadvisor.com sensitive, or confirm the op is truly read).

### HI-02: The augment left 94 ORPHAN descriptors (31 apps) referencing deleted vendored source -- catalog + seed index + provenance are out of sync, and no gate detects it

**STATUS: FIXED.** `runImport` now PRUNES-TO-MATCH: it deletes every `opentabs__*.json` not in the freshly-emitted set, and `feedSeedDescriptors` / `fillProvenance` drop stale opentabs seed slugs + app manifest entries (the 13 hand-only apps + grafana are preserved automatically -- their hand-authored src/tools/*.ts is valid backing, so they are re-emitted). 77 orphans pruned (the authoritative importer-emit count; the review's "94" was the naive `provenance.sourcePath`-existence over-count, which false-positives on kebab/snake op-filename mismatches -- youtube-music underscores -- and on tool.name-vs-filename ops like reddit.delete/teams.invite_to_channel/wikipedia.get_page_summary). New build-failing gate `scripts/verify-no-orphan-descriptor.mjs` (+ `tests/no-orphan-descriptor.test.js`) re-derives the importer's would-emit set and FAILS if any committed descriptor lacks backing -- wired into `validate:extension`. Catalog regenerated (2306 opentabs descriptors; recipe-index 2314; INV-01 IIFE/djb2 intact). Drift gone (manifest 127 apps; seed reddit.list_subreddit_posts orphan dropped).

**File:** `scripts/import-opentabs-catalog.mjs:759-847` (runImport -- writes only, never prunes) + the committed `catalog/descriptors/opentabs__*.json` corpus

**Issue:** The full-source import (commit b9a4b22a) ADDED the real apex-host descriptors but did NOT remove the stale descriptors emitted by the earlier hand-authored Phase-38/39 slices. `runImport` only `writeFileSync`s the freshly-emitted files; there is NO `unlink`/prune of descriptors whose source op no longer exists (confirmed: no delete logic anywhere in the importer). When the vendored slice for an app was swapped from the old hand slice (e.g., `*://www.doordash.com/*`, ops place_order/cancel_order/track_order/...) to the real slice (`*://*.doordash.com/*` -> apex, ops bookmark_store/get_order/update_profile/...), the OLD descriptors were orphaned, not overwritten (different filenames, different host).

Measured on the committed tree:
- **94 orphan opentabs descriptors** (of 2,383) whose `provenance.sourcePath` points to a `.ts` that does NOT exist in `vendor/opentabs-snapshot/` (live=2,289, orphan=94).
- **31 apps drift** between on-disk descriptor count and the `_provenance.json` manifest count (e.g., doordash manifest=11/disk=16, tripadvisor 12/15, ebay 8/13, uber 8/13, ytmusic 0 manifest entry/15 disk under music.youtube.com).
- **All 94 orphan slugs are ALSO in `catalog/descriptors/_fixtures/seed-descriptors.json`** -- the eval/search index -- so stale ops pollute the shipped capability search.
- Orphans include payment/destructive ops: `doordash.place_order`, `walmart.place_order`, `ebay.buy_now`, `instacart.checkout`, `bestbuy.place_order`, `youtube-music.delete_playlist`, etc. (Their hosts happen to remain gated via `*.` patterns, so this is data-integrity, not an Auto-bypass -- EXCEPT the tripadvisor case, HI-01.)

Why HIGH and not a cosmetic data issue: (1) it directly defeats the phase's stated "augment PRESERVES (not clobbers)" guarantee in the opposite direction -- it preserved DEAD data; (2) it is the root cause of HI-01 (the stale www tripadvisor reads mask the live apex write from `checkReadOnlySafeOrigins`); (3) all gates pass, so there is no signal that the committed catalog contains 94 descriptors for ops the source no longer defines; (4) the shipped search index recommends ops that cannot be re-derived/audited.

**Fix:**
1. Re-run a CLEAN import: empty `catalog/descriptors/opentabs__*.json` (and the opentabs slugs in `seed-descriptors.json`) before `runImport`, OR have `runImport` compute the full set of emitted filenames and `unlink` any `opentabs__*.json` not in that set (prune-to-match). Then `feedSeedDescriptors` should likewise drop opentabs seed slugs no longer emitted.
2. Add a CI gate (mirror verify-no-duplicate-stem) that FAILS if any `opentabs__*.json` descriptor's `provenance.sourcePath` does not exist under `vendor/opentabs-snapshot/`, and that asserts disk count == `_provenance.json` manifest count per app. This turns the orphan class into a build failure for every future re-vendor.

## Medium

### MED-01: `checkReadOnlySafeOrigins` only screens the DECLARED class over the committed corpus, so it inherits the orphan/host-form gap by construction

**STATUS: FIXED.** Added `isReadOnlySafeService()` (host-form-agnostic: strips/adds a leading `www.` before the set lookup); `checkReadOnlySafeOrigins` and the commerce disjointness check now both use it, so a brand listed in one host form covers the other emitted form. Pairs with HI-01.

**File:** `scripts/verify-catalog-crosscheck.mjs:225-247`

**Issue:** The guard keys strictly on `d.service` string equality against `READ_ONLY_SAFE_SERVICES`. It has no awareness that one logical brand can emit under two host strings (apex + www), nor that the committed corpus can contain orphan descriptors. As a result the "X is read-only" invariant is only as strong as the exactness of the host strings in the set -- which HI-01 shows is fragile. The commerce backstop avoided this by classifying via `*.`-pattern `classify()` (host-form-agnostic); the read-only-safe guard did not get the same treatment.

**Fix:** Normalize the host before the set check (e.g., also test the `www.`-stripped and `www.`-prefixed forms), and/or drive the read-only-safe set as a roster of BRANDS resolved through the same host-form logic the denylist uses. Pairs with HI-01.

### MED-02: `checkCommerceSensitiveClassified` asserts only ONE host form per brand, relying implicitly on `*.` denylist patterns -- a future EXACT-host denylist entry would create a silent apex gap

**STATUS: FIXED.** `checkCommerceSensitiveClassified` now asserts classify() is sensitive/denied for BOTH the apex AND the www. form of every roster brand. This immediately surfaced a REAL latent gap: opentable/ticketmaster/stubhub/eventbrite were EXACT-host `www.<brand>` so their apex classified SAFE -- WIDENED to apex-suffix `https://*.<brand>` in service-denylist.json (the same conservative discipline as the 18 other commerce brands; covers apex+www+subdomain, no broader than the brand's own origin). The backstop is now robust to a future exact-host pattern narrowing.

**File:** `scripts/verify-catalog-crosscheck.mjs:290-356`

**Issue:** The backstop is currently correct ONLY because every commerce entry in `service-denylist.json` uses the `*.<brand>` suffix form, which covers apex + www. But the roster lists a single host per brand (e.g., `www.amazon.com`, `doordash.com`), so the assertion passes as long as that ONE string classifies sensitive. If a future editor "narrows" a commerce pattern to an EXACT host (e.g., `https://www.amazon.com`, dropping the `*.`) to reduce scope, then the apex `amazon.com` (which several brands DO emit -- ebay.com/walmart.com/target.com/booking.com/airbnb.com/expedia.com/instacart.com/dominos.com/chipotle.com/costco.com/bestbuy.com all emit BOTH forms) would classify SAFE and the backstop would NOT catch it, because the roster checks the www form. The roster does not assert that BOTH emitted forms are covered.

This is latent (today's patterns are all `*.`), hence MEDIUM. But it weakens the backstop's promise of catching "another brand with an emitted-origin-vs-classification mismatch."

**Fix:** In `checkCommerceSensitiveClassified`, for each roster brand assert `classify()` is sensitive/denied for BOTH the apex and the `www.` form (or for every host form the brand actually emits, read from the corpus). That makes the backstop robust to a future EXACT-host narrowing, not just the current `*.` happy path.

## Low

### LO-01: Defensive carve `carveBusinessCodeField` renames a top-level `code` property without verifying the app/op -- the app+op guard lives only at the call site

**STATUS: FIXED.** `carveBusinessCodeField(params, app, opName)` now asserts the exact `(app === 'target' && opName === 'apply_promo_code')` pair INSIDE the function (returns a no-op otherwise), so the rename can only ever apply to target.apply_promo_code regardless of call site. Call site updated to pass app/op. FORBIDDEN set untouched.

**File:** `scripts/import-opentabs-catalog.mjs:386-399` (carveBusinessCodeField) + call site `:714-716`

**Issue:** `carveBusinessCodeField(params)` unconditionally renames ANY top-level `code` property to `promo_code`. The narrowing (`app === 'target' && tool.name === 'apply_promo_code'`) is enforced only at the single call site. The function itself is exported-adjacent build tooling, so a future second call site (or a test) could invoke it on an unintended schema and silently rename a legitimate `code` field, masking a Wall-1 hit that `assertCleanParams` would otherwise catch. The actual eval-ability risk is unchanged today (only one call site, and `assertCleanParams` still runs AFTER), so this is LOW -- a robustness/footgun note, not a live vulnerability.

**Fix:** Pass the app/op into the function and assert the exact pair inside it (defense-in-depth), so the rename can only ever apply to `target.apply_promo_code` regardless of call site:

```js
function carveBusinessCodeField(params, app, opName) {
  if (!(app === 'target' && opName === 'apply_promo_code')) return;
  // ... existing rename ...
}
```
