## Security Section — Headline: No Money-Mover Under Auto

The headline security guarantee of this phase is **VERIFIED on a REAL payment op**, closing the context's stated review correction (the e2e payment-write proof was re-pointed from doordash — which has NO `place_order` at this SHA, the old proof was on a stale hand-authored orphan — to **amazon**, a real hand-only payment write).

**Verified end-to-end on the SHIPPED `opentabs__amazon__place_order.json` descriptor** (`sensitive-write-import-gate.test.js`, 30 passed):

1. `classify(https://www.amazon.com)` → `{ sensitive:true, denied:false }` in the LIVE committed roster (the 39-01 payment screening governs amazon) — the write-gating precondition holds.
2. The real emitted `amazon.place_order` descriptor: `service=www.amazon.com`, `sideEffectClass=write`, `backing=dom` (a money-moving order placement, DOM-only on a sensitive origin).
3. `amazon.place_order` (payment write) on `www.amazon.com` **with NO flag → NOT allow** → `error.code === RECIPE_CONSENT_MUTATING_REQUIRED` (byte-exact, dual-field INV-03). **No order is placed under Auto.**
4. The read sibling `amazon.list_orders` (read) on the same sensitive origin → **allow** (reads run under Auto; the re-gate is write-only — the conservative posture: sensitive origin reads freely, gates every write).
5. WITH the per-origin mutating flag → **allow** (explicit elevation) — the friction is a deliberate consent gate, not a dead end.

**Full-scale payment-op guard** (`verify-catalog-crosscheck.mjs` + `payment-op-guard.test.js`, 20 passed): over the full 2306-descriptor corpus, **NO payment-bearing op is safe-and-API-invocable**. All payment ops are `backing:'dom'` (invocable=false). Dual-path detection (verb family + literal PAYMENT_OP op-name set) catches both `place_order`-style verbs and benign-verb names like `create_order` (PATH 2). A payment op promoted to a backing handler FAILS the build even on a SENSITIVE origin.

**Conservative commerce reconciliation** (the security-critical task): all **32 commerce/payment/travel/marketplace brands classify SENSITIVE on EVERY emitted host form (apex AND www)** and are disjoint from READ_ONLY_SAFE — 0 commerce brands ship safe-under-Auto. So a future op-set change (a brand gaining a real write op) cannot silently make a money-mover writable-under-Auto.
