# 24-01 Plan: Protocol/Envelope Adapter

## Goal

Make `extension/ws/ws-client.js` use PhantomStream-compatible protocol constants and envelope helpers for stream transport frames while preserving current dashboard task/status traffic and the stateless `_lz` compression contract.

## Tasks

1. Add a browser IIFE bundle entry that imports `@full-self-browsing/phantom-stream/protocol` and exposes `globalThis.FSBPhantomStreamProtocol` for the classic service worker.
2. Load the generated protocol bridge in `extension/background.js` after `lib/lz-string.min.js` and before `ws/ws-client.js`.
3. Refactor `ws-client.js` to:
   - resolve `STREAM` and `CONTROL` constants from the protocol bridge with local fallbacks;
   - encode outbound frames through `encodeEnvelope(...)` when available;
   - decode inbound frames through `decodeEnvelope(...)` when available;
   - preserve FSB's guard that compressed envelopes are sent only when the full encoded wire string is smaller than raw JSON;
   - record explicit decode diagnostics for malformed JSON, missing decompressor, decompression failures, and invalid inner JSON.
4. Add/update tests that prove the protocol bundle has no ESM imports, load order is correct, helper calls are wired, and LZ round-trip behavior remains symmetric.
5. Do not change relay room routing, reconnect behavior, remote-control CDP dispatch, or dashboard task/status message semantics in this plan.

## Done When

- `ws-client.js` has a package-backed protocol/envelope seam with safe fallbacks.
- `_lz` remains self-identifying and stateless.
- Non-stream dashboard messages continue through the same `FSBWebSocket.send(type, payload)` envelope.
- RELAY-01 is complete; RELAY-03 has envelope-level evidence but remains open until relay/dashboard-side compression parity is closed.
