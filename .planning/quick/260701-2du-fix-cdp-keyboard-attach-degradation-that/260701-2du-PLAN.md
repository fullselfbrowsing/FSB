<objective>
Fix the CDP keyboard-attach degradation that silently drops trusted keystrokes into cross-origin iframes (e.g. the Stripe CVC frame), blocking checkout completion.

Root cause (confirmed in code): `KeyboardEmulator.attachDebugger` (extension/utils/keyboard-emulator.js) memoizes the in-flight attach in `this.attachPromise`. On a FAILED attach it calls `resolve(false)` (~line 227) but never nulls `attachPromise`. `attachPromise` is only cleared in `detachDebugger` (~line 251) -- which early-returns at ~line 240 (`if (!this.debuggerAttached || !targetTabId) return;`) whenever the attach failed (`debuggerAttached` is false). So a failed attach leaves a cached `false`-resolving promise forever; every later keystroke short-circuits at ~line 212 (`if (this.attachPromise) return await this.attachPromise;`) and returns cached `false`. `sendKeyEvent` then returns failure and the content-script `keyPress` (extension/content/actions.js ~4104-4111) SILENTLY falls back to synthetic `domEvents` (isTrusted:false), which no-op inside cross-origin iframes. Because `keyboardEmulator` is a single background-global singleton, one poisoned promise degrades EVERY tab for the life of the service worker.

Purpose: Unblock trusted keystroke delivery (Stripe CVC and any cross-origin payment/OAuth iframe) by making the keyboard CDP attach self-healing, and stop the content script from reporting a no-op untrusted dispatch as a plain success.

Output: Surgical edits to keyboard-emulator.js (Tier 0 + Tier 1) and content/actions.js keyPress (Tier 2), plus a new Node regression test wired into `npm test`.
</objective>
