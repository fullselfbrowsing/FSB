# Quick Task 260630-u62: Make Costco T1-ready

## Goal

Promote Costco's safe public ecom read subset from DOM/discovery tail to verified T1a handler coverage.

This task must not activate Costco account, shopping-list, cart, checkout, navigation, rendered-search-page, payment, membership, or authenticated digital-account flows. Costco is in the retail/payment sensitive category; only param-driven public no-auth reads with source-reviewable API contracts may become executable here.

## Source Audit

- GOAL: "Make this app costco T1-ready" is covered by adding handler-backed T1a coverage for Costco's safe param-driven product read subset and refreshing readiness/search surfaces.
- STATE invariants: keep the two-tool capability surface, MV3 Wall 1, Wall 2 origin pin, sensitive-origin handling, and fail-closed write posture. No per-app MCP tool, OpenTabs runtime code, direct fetch, or generic cross-origin credential bridge is introduced.
- Phase 44/51: Costco currently has discovery-pending descriptors. This plan promotes only source-reviewable public reads; other Costco rows stay non-executable with honest terminal-state accounting.
- Existing patterns: follow the current quick-task T1 head shape used by `target.js` and `chipotle.js`: dual IIFE/CommonJS handler, extension copy parity, `executeBoundSpec` only, app-specific origin proof, readiness overrides, and focused negative controls.

## T1-ready slugs

- `costco.get_product`
- `costco.get_products`
- `costco.get_product_availability`

## Excluded rows

Do not register these rows in `FsbHandlerCostco`: `costco.add_to_list`, `costco.create_list`, `costco.delete_list`, `costco.geocode_location`, `costco.get_current_user`, `costco.get_list_items`, `costco.get_lists`, `costco.navigate_to_cart`, `costco.navigate_to_checkout`, `costco.navigate_to_product`, `costco.navigate_to_search`, `costco.remove_list_item`, and `costco.search_products`.

Rationale: list/current-user rows use `api.digital.costco.com`, bearer auth, cookies, membership data, or account state; cart/checkout/product/search navigation rows mutate page location through `window.location`; `search_products` depends on the rendered current DOM and has no keyword parameter in its descriptor; `geocode_location` uses a separate public host and is intentionally not included in this focused ecom product-read head.

## Tasks

### 1. Add the Costco public ecom read handler

Files:
- `catalog/handlers/costco.js`
- `extension/catalog/handlers/costco.js`
- `tests/capability-head-handlers.test.js`

Action:
- Create a dual IIFE/CommonJS handler module exporting `FsbHandlerCostco`.
- Use `COSTCO_ORIGIN = "https://www.costco.com"`, `COSTCO_SERVICE = "costco.com"`, `PRODUCT_API = "https://ecom-api.costco.com/ebusiness/product/v1/products/graphql"`, and `INVENTORY_API = "https://ecom-api.costco.com/ebusiness/inventory/v1/inventorylevels/availability/batch"`.
- Register only the three T1-ready slugs listed above as `tier: "T1a"`, `origin: COSTCO_ORIGIN`, `sideEffectClass: "read"`, with params copied from the existing descriptor JSON.
- Build specs only through `ctx.executeBoundSpec`; never call direct `fetch`, `XMLHttpRequest`, `chrome.scripting`, `chrome.tabs`, `chrome.cookies`, `chrome.webRequest`, `window.location`, `document.querySelector`, `getCookie`, `document.cookie`, `localStorage`, or `sessionStorage`.
- Use public no-auth CORS specs: `authStrategy: "none"`, `credentials: "omit"`, `origin: COSTCO_ORIGIN`, `extract: "@"`, and JSON headers. Do not read or send bearer tokens, cookies, CSRF tokens, hashed user IDs, membership data, `Authorization`, or `api.digital.costco.com`.
- Use the public Costco client identifiers already present in `vendor/opentabs-snapshot/plugins/costco/src/costco-api.ts`; treat them as public front-end identifiers, never as secrets, and do not log them.
- Implement these endpoint mappings:
  - `get_product`: POST `PRODUCT_API` with a bounded GraphQL `query` for one numeric `item_number`, `clientId = PRODUCT_CLIENT_ID`, locale `en-us`, and `warehouseNumber = args.warehouse_number || "847"`.
  - `get_products`: POST `PRODUCT_API` with up to 25 numeric item numbers, same headers/body shape as `get_product`.
  - `get_product_availability`: POST `INVENTORY_API` with up to 30 numeric item numbers, `selectedWarehouse: "847-wh"`, `distributionCenters: []`, and the public inventory client headers. Do not read the `WAREHOUSEDELIVERY_WHS` cookie.
- Validate item numbers and warehouse numbers as digit-only strings before building request bodies; fail closed without calling `executeBoundSpec` on invalid input.
- Map response shapes using the vendored mapper intent in `vendor/opentabs-snapshot/plugins/costco/src/tools/schemas.ts`: product item number/name/brand/price/list price/rating/review count/image/buyable/in-stock/program types/membership flag/promotional fields/max quantity, plus inventory online/warehouse/pickup/third-party availability fields.
- Return dual-field `RECIPE_DOM_FALLBACK_PENDING` with `fellBackToDom: true` on missing `ctx.executeBoundSpec`, redirects, 401/403, HTTP errors, GraphQL `errors`, non-object/non-array shapes, missing required response fields, unknown item numbers, or invalid inputs.
- Copy `catalog/handlers/costco.js` byte-for-byte to `extension/catalog/handlers/costco.js`.
- Extend `tests/capability-head-handlers.test.js` with Costco fixtures and assertions: three exported read slugs, excluded rows absent, product and inventory specs use `ecom-api.costco.com` with `origin: https://www.costco.com`, `authStrategy: none`, `credentials: omit`, mapped outputs, invalid-input fallback without execution, HTTP/shape fallback, source banned-token checks, and extension-copy parity.

Verify:

- `node tests/capability-head-handlers.test.js`
