# Quick Task 260531-6n5 -- Stub node:* in offscreen lattice-host bundle (UAT-1 CSP fix)

One-liner: Replaced the esbuild `external: ['node:fs/promises', ...]` array on the
offscreen-lattice-host entry with an inline plugin that resolves every `node:*`
specifier to a local stub module at build time, eliminating all CSP-blocking
imports from the regenerated offscreen ESM bundle (3 -> 0).

## Root Cause

`esbuild.config.js` previously marked the Lattice SDK's `node:fs/promises`,
`node:path`, `node:url`, `node:fs`, and `node:crypto` top-level imports as
external on the offscreen-lattice-host entry. The accompanying comment claimed
tree-shaking would remove the unreachable artifact-storage code paths at minify
time. That claim was wrong on two counts:

1. ESM externals are NOT tree-shaken. esbuild preserves the import specifier
   verbatim in the emitted output as a live `import '...' from 'node:fs/promises'`
   (or similar) statement. The fact that the imported symbols are unreferenced
   does not let esbuild strip the import.
2. Chrome MV3 enforces `script-src 'self'` on offscreen documents. The browser
   parser rejects any `import` statement whose specifier is not a same-origin
   URL, including `node:*` bare specifiers. The offscreen document fails to load
   before the first line of FSB code runs.

This was the second of two UAT-1 sub-failures that prevented the offscreen
lattice-host document from booting. The first (offscreen HTML pointing at
unbundled source) was fixed in quick task 260531-63l; the two together unblock
the offscreen surface end-to-end.

The Phase 5 author's error was assuming "external" implied "tree-shakeable
no-op" for browser-target ESM. It does not. Browser ESM bundles require
build-time stubbing of every node-builtin specifier, even when the importing
module's exports are unreferenced from the entry surface.

## Fix Shape

Replaced the `external: [...]` array on the offscreen-lattice-host entry with
an inline esbuild plugin named `stub-node-builtins`:

- `onResolve({ filter: /^node:/ }, ...)` routes any `node:*` specifier into a
  synthetic `node-stub-ns` namespace, regardless of which specific
  builtin is being imported.
- `onLoad({ filter: /.*/, namespace: 'node-stub-ns' }, ...)` serves a tiny
  inline JS module exporting no-op shims for the fs / path / url surface
  Lattice's artifact-storage references:
  - `default = {}`
  - `join(...p) = p.filter(Boolean).join("/")`
  - `fileURLToPath(u) = String(u)`
  - `mkdir / readFile / readdir / rm / stat / writeFile` -- async no-ops

The plugin is inline (no new npm dependency) and scoped to the
offscreen-lattice-host entry only. The offscreen-stt and
content-canvas-interceptor entries do not import node:*, so they were left
byte-identical.

With the stub module in place, esbuild can dead-code-eliminate the entire
artifact-storage subgraph (it now resolves to inert intra-bundle no-ops with
no external dependencies), so the regenerated bundle is leaner AND CSP-clean.

## Stub Function Name Probe (clarification)

The plan's verify block included a secondary sanity probe
(`grep -cE '(fileURLToPath|mkdir|readFile)' extension/dist/offscreen/lattice-host.js`
must be > 0) intended as "proof the plugin fired." On the regenerated bundle
this probe returns 0 -- but that is the better outcome, not a failure:

- The plugin DID fire (otherwise the 5 distinct `node:*` import specifiers
  would still be present in the output; instead the grep count is 0).
- After the plugin routed every node:* import into the stub module, esbuild's
  dead-code-elimination pass observed that none of the stub exports were
  reachable from the offscreen entry's runtime call graph
  (checkpoint / signer / survivability code paths do not invoke
  artifact-storage), so it stripped the stub symbols entirely.

The CSP-survival criterion (`grep -cE 'node:(fs|path|url|crypto)' returns 0`)
is the load-bearing assertion and it passes. The stub-symbol probe was a
weaker secondary signal that DCE legitimately rendered moot. Result: the
production bundle is leaner AND CSP-clean.
