# Phase 260608-93i Plan 01: Re-do panel auto-collapse safely + per-tab isRunning Summary

Two tab-scoping fixes landed: (Issue A / QT-93i-01) `chrome.action.onClicked` now fires `chrome.sidePanel.setOptions` and `chrome.sidePanel.open` Promises synchronously inside the user-gesture window then awaits `open` first and `setOptions` last — the prior 779bbae2 bug (`await setOptions` before `await open` consuming the gesture token and forcing the popup fallback) is structurally impossible in the new code, and a brand-new `chrome.tabs.onActivated` listener gates `sidePanel.setOptions` per `findActiveAutomationSessionForTab` truthiness. (Issue B / QT-93i-02) sidepanel's module-scope `isRunning` + `currentSessionId` are now a per-tab `Map<tabId, { isRunning, sessionId }>` with explicit `tabId` arguments on `setRunningState` / `setIdleState` / `setErrorState`; the active-tab snapshot mirror keeps the existing readers (`updateSendButtonState`, keydown handler, stopAutomation) working unchanged; `chrome.tabs.onActivated` re-syncs the snapshot after `swapToTabConversation`; `automationComplete` + `session_ended` route the running-state transition by the originating tab (`request.tabId`).

## Anti-regression Note (779bbae2 bug shape)

The new smoke's Part 2.1 strips line + block comments from the extracted `chrome.action.onClicked` body and then asserts that the first `\bawait\s` keyword occurrence falls AFTER both Promise assignments. This is a structural source-order guard: any future code edit that reintroduces the `await setOptions; await open` shape inside `chrome.action.onClicked` will fail Part 2.1 in the test chain (since the first `await` keyword would appear before the `openPromise = ...` assignment). The bug shape is therefore structurally prevented in CI, not just by code review.

Part 2.4 also exercises the popup-fallback path with a rejecting `open()` mock, confirming `chrome.windows.create` is invoked when the side panel API fails after the gesture is intact. The opposite path (Part 2.3b) confirms that when `open()` succeeds the popup fallback is NOT invoked.

### Auto-fixed Issues

**1. [Rule 1 - Bug] Test assertion Part 2.1 needed comment-stripped regex to avoid matching the word "awaited" in the regret-explanation comment.**

- **Found during:** Task 3 (initial standalone run of the new smoke).
- **Issue:** The plan-supplied assertion used `onClickedBody.indexOf('await')` to detect the first `await` occurrence in the chrome.action.onClicked body. The new code's body includes the explanatory comment `// awaited chrome.sidePanel.setOptions BEFORE chrome.sidePanel.open` documenting why the prior bug shape was wrong. The naive `indexOf('await')` matched the `awaited` substring (position 273), which is BEFORE the Promise assignments at positions ~1024 + ~1473. Part 2.1 therefore failed even though the actual code path is correct.
- **Fix:** Updated the assertion to strip `/* ... */` block comments and `// ...` line comments from the extracted body before scanning, and to match the `await` keyword via `\bawait\s` (word-boundary + whitespace, so `awaited` still does NOT match the substring `await` followed by `e`). The semantic intent is unchanged: enforce that both Promise assignments come BEFORE the first runtime `await` keyword.
- **Files modified:** `tests/sidepanel-tab-scoping-fix-redo-smoke.test.js` (Part 2.1 block).
- **Resulting assertion text:** `'Part 2.1 -- both setOptionsPromise + openPromise assignments precede the first await keyword (comment-stripped source-order scan)'`.
- **Commit:** Included in `5f846208` (Task 3 single commit; deviation applied before commit).

6. **Let tab A's task complete while user is on tab B.** Tab B's panel stays collapsed (no flicker, no surprise "completed" UI on the wrong tab). Switch back to tab A and re-open the panel: tab A's conversation log shows the completion bubble; tab A's UI is back to idle (`Ready`).
