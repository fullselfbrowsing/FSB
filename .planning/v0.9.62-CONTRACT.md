**Status:** Locked. Phases 255-260 reference these names verbatim and do not re-derive them.

## Scope and intent

This artifact pins five decisions every downstream phase (255-260) will cite verbatim:

1. The canonical list of MCP **action tools** that carry the new visual-session field bundle (36 tools).
2. The field-bundle key names: `visual_reason` (required string), `client` (required, allowlisted), `is_final` (optional boolean).
3. The badge-allowlist reuse citation pointing at the existing v0.9.36 shared validator (no per-tool duplication; no freeform `client` strings).
4. The three typed-error names: `VISUAL_FIELDS_REQUIRED`, `BADGE_NOT_ALLOWED`, `TOOL_REMOVED`.
5. The complete read-only "do not touch" MCP tool list (15 tools) -- their input schemas remain byte-for-byte unchanged through this milestone.

Phase 254 produces only this artifact -- no code is modified, no tests are added, no MCP server runtime behaviour changes. Dependency rule: the removal of the old explicit `visual_session` start/end tools in Phase 258 lands AFTER Phases 255-257 wire the new contract, so callers always have a working path forward.

The canonical action-tool list and the read-only schema-lock list below were cross-checked against the live `TOOL_REGISTRY` in `mcp/ai/tool-definitions.cjs` on 2026-05-11 and reconciled with REQUIREMENTS.md CONTRACT-01 / CONTRACT-05 via four explicit milestone-level resolutions (see "Resolutions applied" section below).

## Field Bundle

Every action tool in the canonical list above accepts and requires the following field bundle on every MCP input call:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `visual_reason` | string | required | Short human-readable reason shown in the overlay (for example, "Logging in to GitHub"). |
| `client` | string | required | Allowlisted client label. Validated against the shared v0.9.36 server/extension allowlist (see "Badge Allowlist (citation)" below). |
| `is_final` | boolean | optional | When `true`, the visual session clears immediately after the tool's `change_report` resolves. Default: `false`. |

Naming conventions:
- Field keys use `snake_case`, matching existing MCP tool argument conventions (`tab_id`, `session_token`, `change_report`, etc.).
- The `client` field reuses the field name already in use by the v0.9.36 explicit `start_visual_session` tool, minimising migration friction.
- `is_final` is the explicit task-completion signal documented under Phase 257.

Read-only MCP tools do NOT carry this bundle. See "Read-Only Tools (Schema Lock List)" below.

## Badge Allowlist (citation)

The `client` field is validated against the existing v0.9.36 shared server/extension badge allowlist. No new allowlist is created for v0.9.62; no per-tool duplication is permitted; no freeform `client` strings are accepted.

Source-of-truth file (canonical): `extension/utils/mcp-visual-session.js` (lines 4-46)
Constant: `MCP_VISUAL_CLIENT_LABELS` (array of 12 approved client labels)
Validator helpers (same file):
- `normalizeMcpVisualClientLabel(raw)` -- returns canonical label or null
- `isAllowedMcpVisualClientLabel(raw)` -- returns boolean
- `getAllowedMcpVisualClientLabels()` -- returns a copy of the array

MCP-side mirror (same constant declared server-side for stateless validation before forwarding to the extension):
File: `mcp/src/tools/visual-session.ts` (lines 10-27)
Constant: `MCP_VISUAL_CLIENT_LABELS` (same 12 labels)

Adding new badge labels to the allowlist is OUT OF SCOPE for v0.9.62 (governed by the v0.9.36 badge policy; not opened by this milestone).

Deriving `client` automatically from MCP connection / handshake metadata remains the deferred IDENT-FUTURE-01 item; callers continue to send an allowlisted label on every action call.

## Typed Errors

The three typed-error names introduced by this milestone follow the existing v0.9.60 SCREAMING_SNAKE_CASE precedent (see `mcp/src/errors.ts` lines 54-63 for the precedent set: `TAB_NOT_OWNED`, `AGENT_CAP_REACHED`, `TAB_INCOGNITO_NOT_SUPPORTED`, `TAB_OUT_OF_SCOPE`, etc.).

| Error name | Surfaces | Phase | Body intent |
|------------|----------|-------|-------------|
| `VISUAL_FIELDS_REQUIRED` | Schema layer; rejects action calls missing `visual_reason` or `client` before the underlying action runs. | 255 | "The visual-session field bundle is required on action tools. Supply `visual_reason` (string) and `client` (allowlisted label). See the v0.9.62 contract recipe." |
| `BADGE_NOT_ALLOWED` | Schema layer; rejects `client` values not on the shared v0.9.36 allowlist. | 255 | "The `client` value is not on the trusted v0.9.36 badge allowlist. Allowed values: <enumerated from `getAllowedMcpVisualClientLabels()`>." |
| `TOOL_REMOVED` | Removal layer; replaces dispatch of removed `start_visual_session` / `end_visual_session` tool names. | 258 | "The explicit `visual_session` start/end tools were removed in v0.9.62. Use the implicit contract: required `visual_reason` / `client` on action tools, sliding 60s window, `is_final: true` for early clear. See <CHANGELOG section + mcp/README visual-session section>." |

The error name strings (`VISUAL_FIELDS_REQUIRED`, `BADGE_NOT_ALLOWED`, `TOOL_REMOVED`) are pinned exactly as written. Downstream phases reference them verbatim and do not introduce variants.

- `report_progress`, `complete_task`, `partial_task`, `fail_task` are task-lifecycle reporting hooks; do not mutate DOM and emit signal events only.
- `wait_for_element` and `wait_for_stable` observe page state (await predicates / DOM stability); reclassified as read-only for this milestone per the user-level resolution.
