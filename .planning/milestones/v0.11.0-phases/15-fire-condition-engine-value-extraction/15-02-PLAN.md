<design_resolution>
RESOLVED (PATTERNS.md open question - concurrent-arm cap race): `agent-registry.js` serializes its cap-check+insert inside `withRegistryLock` (:181, a 4-line promise-chain mutex) because the active count is a synchronous heap `Set.size`. `trigger-manager.armTrigger` reads the active count ASYNC from `listArmedSnapshots()` (storage), so a naive read-count-then-write has a genuine TOCTOU race: two concurrent arms both read active=7 under cap=8 and both proceed to 9. DECISION: port the `withRegistryLock` mutex as a module-scope `_withArmLock(fn)` (a single `var _armChain = Promise.resolve();` that each `armTrigger` chains onto) and run the `listArmedSnapshots()` read + the cap compare + the `FsbTriggerLifecycle.armTrigger(snapshot)` write INSIDE that lock, so concurrent arms serialize and the cap is enforced atomically. This is mandatory for the N-concurrent cap test to pass. The lock guards ONLY armTrigger (arm is not the hot path); `evaluate()` is lock-free and pure.
</design_resolution>
