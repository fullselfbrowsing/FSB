<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| reloaded page -> content triggerRead | A reload can land on a different page class than the armed page: login, CAPTCHA, restricted, or challenge. |
| content blocked response -> persisted trigger snapshot | Blocked signals must become durable attention state for later status/reporting tools, not a fire or silent no-op. |
| service worker -> content pulse route | After reload, the watched page has a fresh content context and needs the existing pulse reasserted only if the trigger remains armed. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-02 | Spoofing / Tampering | challenge/login/CAPTCHA page false fire | mitigate | `triggerRead` returns `TRIGGER_PAGE_BLOCKED` before selector/value reads; background writes `status:'blocked'` and never stages blocker text or calls `handleTriggerAlarm` for blocked pages. |
| T-17-04 | Tampering | stale selector / missing/wrong element | mitigate | Plan 17-02 and Plan 17-03 handle `ELEMENT_NOT_FOUND` as needs_attention. This plan adds blocker classification so wrong-page content cannot masquerade as the watched element. |
| T-17-01 | Tampering / Denial of Service | cross-agent reload before ownership validation | mitigate | Plan 17-03 ownership gate remains in front of all reload/read/pulse behavior; final tests keep no-focus/no-active-tab invariants green. |
| T-17-03 | Denial of Service | reload hammering | mitigate | Pulse restart happens only after a successful non-terminal read; the next poll still uses Plan 17-01 floor-safe scheduling. Blocked/attention states do not schedule immediate reload loops. |
</threat_model>
