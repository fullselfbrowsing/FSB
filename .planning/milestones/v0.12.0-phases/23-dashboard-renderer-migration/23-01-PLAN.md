# 23-01 Plan: Shared Renderer Wrapper

## Goal

Create the shared browser-global PhantomStream viewer wrapper and bundle/load strategy that both static and Angular dashboards will consume.

## Tasks

1. Add a wrapper entry that imports `createViewer`, renderer helpers, and protocol constants from `@full-self-browsing/phantom-stream`.
2. Expose `window.FSBPhantomStreamViewer` with a small FSB-facing API:
   - create/destroy viewer handle;
   - dispatch `STREAM.*` messages into viewer transport;
   - map viewer `CONTROL.START` to a host callback for existing `dash:dom-stream-start`;
   - expose viewport mapping helpers needed by remote-control overlay geometry;
   - surface state/health events without leaking mirrored content.
3. Add an esbuild entry that emits `showcase/js/phantom-stream-viewer.js`.
4. Load the generated bundle in `showcase/dashboard.html`.
5. Update Angular asset/script loading so `phantom-stream-viewer.js` is available on the dashboard route from `assets/phantom-stream-viewer.js`.
6. Add focused tests for wrapper export shape, bundle generation, static script order, Angular asset copy config, and no direct package imports in Angular component code.

## Verification

```bash
npm run build
node tests/phantom-stream-dashboard-viewer-bundle.test.js
node tests/phantom-stream-exports.test.js
npm run validate:extension
git diff --check
```

## Done When

- Both dashboard surfaces can access the same `window.FSBPhantomStreamViewer` global.
- The wrapper has no dashboard-specific state-machine decisions.
- No static or Angular renderer code has been migrated yet except loading the wrapper.
