# 22-02 Plan: Capture Core Migration

## Goal

Replace `dom-stream.js` bespoke snapshot and MutationObserver ownership with a thin adapter around PhantomStream capture while keeping existing dashboard payload compatibility.

## Tasks

1. Rewrite `extension/content/dom-stream.js` so lifecycle methods call a PhantomStream capture handle.
2. Implement the transport adapter from `STREAM.*` package messages to existing FSB background actions:
   - `STREAM.SNAPSHOT` -> `domStreamSnapshot`
   - `STREAM.MUTATIONS` -> `domStreamMutations`
   - `STREAM.SCROLL` -> `domStreamScroll`
   - `STREAM.OVERLAY` -> `domStreamOverlay`
   - `STREAM.DIALOG` -> `domStreamDialog`
   - `STREAM.READY` -> `domStreamReady`
3. Add a temporary legacy identity bridge that stamps `data-fsb-nid` into snapshot HTML and add-op HTML from PhantomStream `nodeIds` sidecars.
4. Preserve FSB `domStreamResume` behavior by forcing a fresh capture start rather than package `resume()`.
5. Keep `FSB.domStream` test/debug accessors focused on adapter state instead of exposing old serializer internals.

## Verification

```bash
node tests/phantom-stream-capture-adapter.test.js
node tests/dashboard-runtime-state.test.js
node tests/dom-stream-perf.test.js
node tests/dashboard-stream-readiness-ping.test.js
```

## Done When

- `dom-stream.js` no longer contains local `serializeDOM`, `processMutationBatch`, or direct `new MutationObserver` ownership.
- Existing dashboards can still consume snapshot/mutation payloads before Phase 23.
- Start, stop, pause, resume, scroll, and stale-flush count behavior are adapter-tested.
