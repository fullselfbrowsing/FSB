## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| extension SW <-> chrome.storage.local | The queue + watermark + opt-out + active-agents-count live here. Storage is in-process to the user's browser profile -- trusted, but corruptible (DevTools storage editor; enterprise policy). The module re-validates every read defensively. |
| extension SW -> https://full-selfbrowsing.com/api/telemetry/events | Outbound JSON over HTTPS. The 9-field allowlist is the trust boundary -- the server receives ONLY what the collector chose to emit. The endpoint is unauthenticated (anonymous telemetry) so the server has no way to verify the client identity beyond `install_uuid`, and that field is itself an opaque anonymous handle. |
| extension SW <-> fsbUsageData rows (written by mcp-metrics-recorder.js) | The collector reads rows produced by Phase 271. A compromised/buggy recorder could write attacker-influenced row contents (e.g. via tool name spoofing). The collector mitigates by NEVER copying `tool`, `cost_usd`, `pricing_confidence`, `token_source` into the outbound payload -- only `client`, `model`, `tokens_in`, `tokens_out`, `ts` are referenced, and `client` + `model` are passed through to the server (which applies its own allowlist per Phase 273 / INGEST-09). |
| MCP agent register/release (Task 2) -> fsbActiveAgentsCount | Counter writes happen inside the dispatcher chokepoint. A bridge that issues malformed agent:release messages cannot drive the counter negative (clamp-to-0); cannot inflate beyond the actual register count under normal operation (cap is enforced upstream by AgentRegistry's existing cap check). |
