### C. JSON-load strategy for the JS module

Two paths -- pick the synchronous-fallback option:

1. **Browser/SW path**: `fetch(chrome.runtime.getURL('utils/mcp-pricing-data.json'))` at module-load time. Cache the resolved Promise on `globalThis.fsbMcpPricing._dataPromise`. The resolver awaits this promise on its first call; subsequent calls reuse the cached data ref synchronously.
2. **Node-test path**: `require('./mcp-pricing-data.json')` works natively under Node CommonJS. Detect via `typeof require === 'function' && typeof chrome === 'undefined'` and load synchronously; expose `_loadPricingData(data)` test seam so the test harness can also inject custom JSON for the unknown-path test.

The resolver function signature `estimateMcpCost(input)` is SYNCHRONOUS -- it must not return a Promise. To guarantee this in the SW, the data load runs at module init via a top-level IIFE that calls `fetch().then(data => { _data = data; })`. Phase 271 imports the module after Phase 269's `install-identity.js` and before `mcp-tool-dispatcher.js` runs any dispatch -- by then the fetch has resolved. As a defensive fallback, if the resolver is called before data load completes, it returns `{cost: null, source: 'unknown', model_used: null, pricing_confidence: null, pricing_source_date: '2026-05-14'}` (same as the no-data-found path -- never throws).

### E. Resolver contract (EXACT shape -- CONTEXT.md decisions)

```typescript
type McpPricingResult = {
  cost: number | null;
  source: 'lookup' | 'fallback' | 'unknown';
  model_used: string | null;
  pricing_confidence: 'HIGH' | 'MEDIUM' | 'LOW' | 'fallback' | null;
  pricing_source_date: '2026-05-14';
};

function estimateMcpCost(input: {
  client?: string | null;
  model?: string | null;
  tokensIn?: number | null;
  tokensOut?: number | null;
}): McpPricingResult;
```

Four resolver paths (CONTEXT.md §decisions §"Fallback policy"):

1. **lookup**: `model` is provided AND is a key in MCP_MODEL_PRICING -> compute cost from the row, source='lookup', model_used=model, pricing_confidence=row.confidence.
2. **fallback**: `model` missing/null/unknown BUT `client` is in MCP_CLIENT_DEFAULT_MODEL AND its default model is in MCP_MODEL_PRICING -> compute cost using default model, source='fallback', model_used=default_model, pricing_confidence='fallback' (literal sentinel string -- NOT the model's own HIGH/MEDIUM/LOW).
3. **unknown**: neither matches -> cost=null, source='unknown', model_used=null, pricing_confidence=null.
4. **missing tokens**: at ANY of the three paths above, if tokensIn or tokensOut is null / undefined / NaN / Infinity / non-number, return cost=null but preserve model_used + source + pricing_confidence as far as the lookup got. THIS IS A LEGITIMATE CASE (telemetry rows where the upstream tool didn't report tokens).

`pricing_source_date: '2026-05-14'` is attached to ALL FOUR return shapes.

NEVER throws. Wrap the whole body in try/catch; catch-all returns the canonical `{cost: null, source: 'unknown', model_used: null, pricing_confidence: null, pricing_source_date: '2026-05-14'}`.

Cost arithmetic: `cost = (tokensIn / 1_000_000) * input_per_mtok + (tokensOut / 1_000_000) * output_per_mtok`.
