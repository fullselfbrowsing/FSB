## How this file gets used

- **At plan-time:** When FSB plans a new phase, the planner reads the frontmatter package/version/integrity fields to ground "what Lattice surface FSB depends on right now."
- **At execute-time:** When a phase changes the Lattice dependency, the executor records the resulting source tag/commit or local SHA in this file via a new row and updates the active frontmatter fields.
- **At verify-time:** `tests/lattice-public-package.test.js` cross-checks `.planning/LATTICE-PIN.md`, `package.json`, `package-lock.json`, installed `node_modules/lattice`, and the Lattice CLI. Legacy smoke tests still exercise the imported runtime surface end-to-end.

## Schema notes

- `current_lattice_source`: `npm` for the active public-package model.
- `current_lattice_package` / `current_lattice_version` / `current_lattice_alias`: the exact runtime package FSB imports.
- `current_lattice_integrity`: registry tarball integrity copied from `package-lock.json`.
- `current_lattice_sha`: 40-char source commit for the active public tag, or historical local clone SHA for pre-public rows.
- `current_branch`: source branch for the active package tag.
- Per-phase rows are append-only; closed phases are not edited retroactively.
- `Lattice work touched` cells are summary-form; the source of truth is the Lattice commit body (grep for `Ref: FSB v0.10.0-attempt-2 Phase N` inside `cd lattice && git log` to find a phase's commits).
